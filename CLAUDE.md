## Project Overview

A digital asset manager designed for large collections of images and videos (terabytes across multiple offline storage devices). Key design goals:

- Content-addressable storage for originals (SHA-based, since originals like RAW files are immutable)
- Text-based metadata in sidecar files
- Grouping of asset variants (RAW/JPEG, different processing versions)
- Deduplication of identical files
- Transparent file relocation across storage devices
- Management of processing recipes (CaptureOne, Photoshop, etc.)
- Location-independent navigation and retrieval

## Technology

- **Language**: Rust
- **Platforms**: macOS, Linux, Windows
- **Interface**: CLI-first (`dam` command)
- **Catalog**: SQLite (cache/index), YAML sidecar files (source of truth)
- **Key crates**: clap, sha2, serde, rusqlite, kamadak-exif, quick-xml, regex, image, imageproc, ab_glyph, lofty, uuid, axum, askama, tokio, tower-http, toml, glob-match; **optional (ai feature)**: ort, ndarray, tokenizers
- **External tools**: dcraw/libraw (RAW previews), ffmpeg (video thumbnails), curl (AI model download)

**Import behavior**:
- **Stem-based auto-grouping**: Files sharing the same filename stem in the same directory are grouped into one Asset during import. RAW files take priority as the primary variant (defining asset identity and EXIF data). Additional media files become extra variants on the same asset.
- **Recipe handling**: Processing sidecars (`.xmp`, `.cos`, `.cot`, `.cop`, `.pp3`, `.dop`, `.on1`) are attached as Recipe records to the primary variant rather than imported as standalone assets. Recipes are identified by location `(volume_id, relative_path)` rather than content hash, so re-importing after external edits updates in place instead of creating duplicates.
- **Standalone recipe resolution**: When a recipe file is imported without a co-located media file (e.g., importing `DSC_001.xmp` after `DSC_001.nef` was already imported), the system finds the parent variant by matching the filename stem and directory, and attaches the recipe to it.
- **XMP metadata extraction**: When an `.xmp` sidecar is attached as a recipe, its contents are parsed (via `xmp_reader` module) and merged into the asset/variant. Keywords (`dc:subject`) become asset tags (deduplicated), `dc:description` sets the asset description (if not already set), `xmp:Rating` is promoted to `asset.rating` (first-class `Option<u8>` field), `xmp:Label` is promoted to `asset.color_label` (first-class `Option<String>` field, validated against the 7-color set: Red, Orange, Yellow, Green, Blue, Pink, Purple), and `dc:creator`, `dc:rights` are stored in the primary variant's `source_metadata`. Rating and label are also kept in `source_metadata` for provenance. EXIF takes precedence for any overlapping `source_metadata` keys. On re-import with changed content, XMP metadata is re-extracted: description is overwritten, rating is overwritten, color label is overwritten, keywords are merged, and source_metadata keys are overwritten.
- **Embedded XMP extraction**: JPEG and TIFF files can contain XMP metadata embedded in their binary structure (APP1 marker for JPEG, IFD tag 700 for TIFF). During import, the `embedded_xmp` module extracts this metadata and merges it via `apply_xmp_data()`. Precedence chain: EXIF (highest, direct assignment) > embedded XMP (middle, `or_insert`) > sidecar `.xmp` (lowest, `or_insert`). For tags, all sources merge (union). For description/rating/label, the first source that provides a value wins. Supported formats: `.jpg`/`.jpeg`/`.tif`/`.tiff`; all other extensions return empty immediately (zero I/O). This captures keywords, ratings, descriptions, and labels from CaptureOne/Lightroom exports that embed XMP directly in the file.
- **XMP write-back**: Rating, tag, description, and color label changes made in the DAM (via CLI `dam edit --rating`/`--description`/`--label`, `dam tag`, web UI stars, web UI label dots, web UI tag add/remove, or web UI inline description editing) are automatically written back to `.xmp` recipe files on disk. Rating uses `xmp_reader::update_rating()` (string-based find/replace of `xmp:Rating`). Label uses `xmp_reader::update_label()` (same pattern: find/replace `xmp:Label` attribute or element; clearing removes the attribute/element entirely rather than setting to empty). Tags use `xmp_reader::update_tags()` with operation-level deltas: added tags are inserted into `dc:subject`/`rdf:Bag`, removed tags are deleted — tags added independently in CaptureOne are preserved. Description uses `xmp_reader::update_description()`: replaces text in existing `dc:description`/`rdf:Alt`/`rdf:li` block, removes the block when clearing, or injects a new block when adding to an XMP file without one. If no `dc:subject` or `dc:description` block exists, one is injected (with `xmlns:dc` namespace and self-closing `rdf:Description` conversion if needed). Special characters in tags and descriptions are XML-escaped. Empty description strings are normalized to `None` (clear) in both CLI and web UI paths. After writing, the file is re-hashed via `ContentStore::hash_file()` and the recipe's `content_hash` is updated in both SQLite catalog and YAML sidecar. Offline volumes and missing files are silently skipped with `eprintln` warnings. This enables bidirectional sync with tools like CaptureOne.
- **Duplicate location tracking**: When a file's content hash already exists, the new file location is added to the existing variant rather than silently skipping. Only truly skips when the exact same location (volume + path) is already recorded.
- **Preview generation**: During import, previews are generated for each variant. Standard image formats use the `image` crate (800px JPEG thumbnails); RAW files use `dcraw` or `dcraw_emu` (LibRaw); videos use `ffmpeg`. Non-visual formats (audio, documents, unknown) get an info card — an 800x600 JPEG showing file metadata (name, format, size, and audio-specific properties like duration/bitrate extracted via `lofty`). Info cards are also generated as a fallback when external tools are missing for RAW/video files. Info card rendering uses `imageproc`/`ab_glyph` with an embedded DejaVu Sans font. Previews are stored in `previews/<hash-prefix>/<hash>.jpg`. Preview failure never blocks import. **EXIF auto-orientation**: EXIF orientation tags are read during preview generation and applied automatically (rotation + flip) so previews display correctly regardless of camera orientation. **Manual rotation**: `preview_rotation` field on Asset (persisted in sidecar YAML + SQLite) applies additional rotation on top of EXIF auto-orientation. Web UI "Rotate" button on detail page cycles 90° CW (0→90→180→270→0), regenerating both regular and smart previews. Preview HTTP responses include `Cache-Control: no-cache` so browsers revalidate after rotation.
- **Export-based preview selection**: Display logic prefers Export > Processed > Original variant previews (skips Sidecar). Within the same role, standard image formats (jpg/jpeg/png/tiff/tif/webp) are preferred over RAW, with file size as a tiebreaker. Implemented via `best_preview_index()` / `best_preview_index_details()` in `src/models/variant.rs`. Used by `dam show`, web UI asset detail page, and `generate-previews` catalog mode. `generate-previews --upgrade` regenerates previews for assets where a better variant exists.
- **Denormalized asset columns**: The `assets` table caches three values computed at write time to avoid expensive JOINs during search: `best_variant_hash` (content hash of the best display variant), `primary_variant_format` (identity format: Original+RAW first, then Original+any, then best variant), `variant_count` (number of variants). Browse grid joins `assets` directly to the best variant (one row per asset, no GROUP BY). Browse cards show the primary format (e.g. NEF) and a variant count badge (e.g. "3v"). Computed by `compute_best_variant_hash()` and `compute_primary_format()` in `src/models/variant.rs`. Updated via `insert_asset()` and `update_denormalized_variant_columns()`. Backfilled during schema migration.
- **Show command**: Displays variants, attached recipes, and preview status for an asset.
- **Edit command**: `dam edit <asset-id> [--name <name>] [--description <text>] [--rating <1-5>] [--label <color>] [--date <YYYY-MM-DD>] [--clear-name] [--clear-description] [--clear-rating] [--clear-label] [--clear-date]`. Sets or clears asset name, description, rating, color label, and creation date from the CLI. At least one flag is required. Updates both sidecar YAML and SQLite catalog. Empty description strings (`--description ""`) are normalized to `None` (same as `--clear-description`). `--label` accepts case-insensitive color names (Red, Orange, Yellow, Green, Blue, Pink, Purple) and validates/canonicalizes to title-case. `--date` accepts ISO date or datetime strings. Supports `--json` for structured output. Uses `QueryEngine::edit()` with `EditFields` (triple-option pattern: `None` = no change, `Some(None)` = clear, `Some(Some(x))` = set). Description, rating, and label changes trigger XMP write-back (see **XMP write-back**).
- **Delete command**: `dam delete <ASSET_IDS...> [--apply] [--remove-files]`. Removes assets from the catalog. Default is report-only mode. `--apply` executes deletion (asset rows, variants, file locations, recipes, previews, sidecar YAML, collection memberships, stack membership). `--remove-files` (requires `--apply`) deletes physical files from disk. Reads from stdin when no IDs given (`dam search -q "orphan:true" | dam delete --apply`). Supports asset ID prefix matching, `--json`, `--log`, `--time`.
- **Export command**: `dam export <QUERY> <TARGET> [--layout flat|mirror] [--symlink] [--all-variants] [--include-sidecars] [--dry-run] [--overwrite]`. Copies files matching a search query to a target directory. Default exports the best variant per asset (Export > Processed > Original) in flat layout. `--layout flat` (default) places all files in the target root; filename collisions resolved by appending `_<8-char-hash>` before the extension. `--layout mirror` preserves source volume-relative directory structure; multi-volume assets get a `<volume-label>/` prefix. `--symlink` creates symlinks instead of copies (unix: `std::os::unix::fs::symlink`, windows: `std::os::windows::fs::symlink_file`). `--all-variants` exports every variant; default is best variant only. `--include-sidecars` also copies recipe files (.xmp, .cos, etc.). `--dry-run` reports the plan without writing files. `--overwrite` re-copies even if the target already has a matching hash; default skips matching files. Files are integrity-verified via SHA-256 after copy (uses `ContentStore::copy_and_verify()`). Duplicate content hashes are deduplicated in the plan. Offline volumes are skipped with warnings. Supports `--json` (`ExportResult` with `assets_matched`, `files_exported`, `files_skipped`, `sidecars_exported`, `total_bytes`, `errors`, `dry_run`), `--log`, `--time`.
- **Import `--volume` flag**: `dam import [--volume <label>] <PATHS...>` — when provided, uses the specified volume instead of auto-detecting from the first path. Useful when auto-detection picks the wrong volume.
- **Import `--dry-run` flag**: `dam import [--dry-run] <PATHS...>` — shows what would be imported without writing to catalog, sidecar, or disk. Files are still hashed (to detect duplicates) but no assets, variants, recipes, or previews are created. Reports the same counters as a real import (imported, skipped, locations added, recipes attached/updated). Supports `--json` (includes `dry_run: true` field) and `--log` for per-file details.
- **Import `--smart` flag**: `dam import [--smart] <PATHS...>` — generates smart previews (2560px JPEG) alongside regular thumbnails during import. Can be enabled permanently via `[import] smart_previews = true` in `dam.toml`.
- **Import `--add-tag` flag**: `dam import [--add-tag <TAG>]... <PATHS...>` — adds tags to every imported asset. Repeatable. Merged with `[import] auto_tags` and XMP tags, deduplicated.
- **Import `--auto-group` flag**: `dam import [--auto-group] <PATHS...>` — after normal import, runs auto-grouping scoped to the "neighborhood" of imported files. Neighborhood is computed by going up one directory level from each imported file's directory to get "session roots," then finding all catalog assets on the same volume under those session roots. This avoids catalog-wide false positives from restarting camera counters (e.g., `DSC_0001` from different shoots). Uses the same fuzzy prefix matching as `dam auto-group`. Combines with `--dry-run` (auto-group also runs in dry-run mode). With `--json`, output includes an `auto_group` key with the full `AutoGroupResult`. Skips auto-group phase if nothing was imported (`imported == 0 && locations_added == 0`).
- **Generate-previews command**: `dam generate-previews [PATHS...] [--asset <id>] [--volume <label>] [--include <group>] [--skip <group>] [--force] [--upgrade]`. Without PATHS, iterates all catalog assets (optionally filtered by `--asset` or `--volume`). With PATHS, resolves files on disk and looks up their variants in the catalog. `--include`/`--skip` filter by file type group. `--force` regenerates existing previews. `--upgrade` regenerates previews for assets where a better variant (export/processed) exists — skips assets where the best variant is already the primary. In catalog mode, selects the best variant per asset using `best_preview_index()` (Export > Processed > Original, image formats preferred, file size tiebreak).
- **Group command**: `dam group <VARIANT_HASHES...>`. Merges variants (identified by content hashes) into a single asset. Picks the oldest asset as the target. Donor variants with role `original` are re-roled to `export` in both sidecar YAML and SQLite catalog. Tags and recipes from donor assets are merged into the target. Donor assets are deleted after merging.
- **Auto-group command**: `dam auto-group [QUERY] [--apply]`. Groups assets by filename stem using fuzzy prefix matching across the catalog (or scoped by an optional search query). Without `--apply`, runs in report-only mode (dry run). **Fuzzy prefix matching**: two stems match if the shorter is a prefix of the longer and the next character in the longer string is non-alphanumeric (a separator like `-`, `_`, ` `, `(`). This handles the common case where export tools (CaptureOne, Lightroom) append suffixes to the original filename: `Z91_8561.ARW` matches `Z91_8561-1-HighRes-(c)_2025_Thomas Herrmann.tif`. The separator boundary check prevents false positives (`DSC_001` does not match `DSC_0010`). Stems are compared case-insensitively. Chains resolve to the shortest root: `Z91_8561`, `Z91_8561-1`, `Z91_8561-1-HighRes` all group under `Z91_8561`. Within each group, the target asset is selected by: (1) prefer asset with a RAW variant (`is_raw_extension()`), then (2) oldest by `created_at`. Donor variants are re-roled from `original` to `export` (see **Group command**). Supports `--json` (full `AutoGroupResult` with `groups`, `total_donors_merged`, `total_variants_moved`, `dry_run`), `--log` (per-group lines to stderr). Web UI: "Group by name" button in the batch toolbar (`POST /api/batch/auto-group` with `{asset_ids}`) — confirm dialog, then merges selected assets by stem.
- **Relocate command**: Copies or moves all files of an asset (variants + recipes) to a target volume. `dam relocate <asset-id> <target-volume> [--remove-source] [--dry-run]`. Without `--remove-source`, files are copied and the asset gains additional locations. With `--remove-source`, source files are deleted after verified copy. `--dry-run` shows what would happen without making changes. Preserves relative paths on the target volume. Verifies file integrity via SHA-256 after copy.

- **Update-location command**: Updates a file's catalog path after it was manually moved on disk. `dam update-location <asset-id> --from <old-path> --to <new-path> [--volume <label>]`. `--to` must be an absolute path to the file's current location. `--from` can be absolute or volume-relative. Volume is auto-detected from `--to` via `find_volume_for_path()`, or specified via `--volume`. Verifies the file at `--to` has the same content hash as the catalog record (safety check). Updates both SQLite catalog and sidecar YAML. Handles both variant file locations and recipe file locations. Supports `--json` (via global flag).

- **Verify command**: Re-hashes files on disk and compares against stored content hashes to detect corruption or bit rot. `dam verify [PATHS...] [--volume <label>] [--asset <id>] [--max-age <DAYS>] [--force]`. Without arguments, verifies all file locations on all online volumes. With paths, verifies specific files/directories. `--volume` limits to a specific volume; `--asset` limits to a specific asset. `--max-age <DAYS>` skips files verified within the given number of days (incremental verify). `--force` overrides `--max-age` and re-verifies everything. Default `max_age_days` configurable via `[verify]` in `dam.toml`. Updates `verified_at` timestamps on successful verification (persisted to both SQLite catalog and sidecar YAML for variant and recipe locations). Exits with code 1 if any mismatches are found. Recipe files that have been modified externally are reported as "modified" (not "FAILED") and do not trigger exit code 1 — their stored hash is updated to reflect the new content.

- **Sync command**: Reconciles catalog with disk reality after external tools move, rename, or modify files. `dam sync <PATHS...> [--volume <label>] [--apply] [--remove-stale]`. Without `--apply`, runs in report-only mode (safe default). With `--apply`, updates catalog and sidecar files for moved files and modified recipes. `--remove-stale` (requires `--apply`) removes catalog location records for confirmed-missing files. Detects: unchanged files (hash matches at expected path), moved files (known hash at new path, old path gone), new files (unknown hash), modified recipes (same path, different hash), missing files (catalog location but file gone from disk). New files are reported but not auto-imported — user runs `dam import` separately. Supports `--json`, `--log`, `--time` flags.

- **Refresh command**: Re-reads metadata from changed sidecar/recipe files without scanning media files. `dam refresh [PATHS...] [--volume <label>] [--asset <id>] [--dry-run] [--media]`. Without arguments, checks all recipe locations on all online volumes. With paths, scans recipe files under given paths. `--volume` limits to a specific volume; `--asset` limits to a specific asset's recipes. For each recipe, compares on-disk hash to stored hash — if changed, re-extracts XMP metadata (keywords, rating, description, color label) and updates catalog and sidecar. Non-XMP recipes (COS, pp3, etc.) get their hash updated but no metadata extraction. `--media` additionally scans JPEG/TIFF variant files and re-extracts their embedded XMP metadata (keywords, rating, description, label, creator, rights) using `reapply_xmp_data()` overwrite semantics — useful for retroactively extracting embedded XMP from files imported before the feature existed, or after external tools edit the embedded metadata. `--dry-run` reports what would change without applying. Reuses `apply_modified_recipe()` internally. Supports `--json`, `--log`, `--time` flags.

- **Cleanup command**: Scans all file locations and recipes across online volumes, checking for files that no longer exist on disk. `dam cleanup [--volume <label>] [--list] [--apply]`. Without `--apply`, runs in report-only mode (safe default). With `--apply`, performs three passes: (1) removes stale location records and recipe records for missing files from catalog and sidecar YAML, (2) deletes orphaned assets (assets where all variants have zero file_locations) along with their recipes, variants, catalog rows, and sidecar YAML files, (3) removes orphaned preview files (previews whose content hash no longer matches any variant in the catalog). `--volume` limits stale-location scanning to a specific volume (otherwise checks all online volumes). `--list` prints stale entries to stderr (unlike `--log` which prints all entries including ok). Skips offline volumes with a note. Checks both variant file locations and recipe file locations. Report-only mode predicts orphaned assets and previews that would result from removing stale locations. Supports `--json`, `--log`, `--time` flags.

- **Stats command**: Shows catalog statistics. `dam stats [--types] [--volumes] [--tags] [--verified] [--all] [--limit N]`. Without flags, shows overview only (assets, variants, recipes, volumes, total size). `--types` adds asset type and format breakdown. `--volumes` adds per-volume details (assets, size, directories, verification). `--tags` shows tag usage frequencies. `--verified` shows verification health. `--all` enables all sections. `--limit N` controls top-N lists (default 20). Supports `--json` for structured output.

- **Embed command** (feature-gated `--features ai`): `dam embed [--query <Q>] [--asset <id>] [--volume <label>] [--model <id>] [--force]`. Batch-generates image embeddings for visual similarity search without tagging. Requires at least one scope filter. For each asset: finds best image (smart preview > regular preview > original), encodes with SigLIP, stores in `embeddings` table. Skips assets that already have an embedding (override with `--force`). Reports embedded/skipped/error counts. Supports `--json`, `--log`, `--time`. Web UI "Suggest tags" and batch "Auto-tag" also store embeddings opportunistically.
- **In-memory embedding index**: `EmbeddingIndex` in `embedding_store.rs` loads all embeddings for a model into a contiguous `Vec<f32>` buffer on first similarity query, then caches in `AppState` (via `std::sync::RwLock`). Search uses dot product (SigLIP embeddings are L2-normalized) with a `BinaryHeap` min-heap for top-K selection. At 100k assets (~300 MB), search completes in <10ms. Updated in-place when new embeddings are stored via any web code path.
- **Faces command** (feature-gated `--features ai`): Face detection, recognition, and people management. Uses YuNet ONNX model for face detection (bounding boxes, landmarks, confidence) and ArcFace for 512-dim face embeddings. Models downloaded via `dam faces download`. Subcommands:
  - `dam faces detect [--query <Q>] [--asset <id>] [--volume <label>] [--apply]` — detect faces in images, store bounding boxes, embeddings, and generate 150×150 JPEG crop thumbnails in `faces/<prefix>/<face_id>.jpg`. Reports faces found per asset. Supports `--json`, `--log`, `--time`.
  - `dam faces cluster [--query <Q>] [--asset <id>] [--volume <label>] [--threshold <F>] [--apply]` — group similar face embeddings into unnamed person groups using greedy single-linkage clustering. Default threshold 0.5 (configurable via `[ai] face_cluster_threshold`). Scope filters limit which faces are clustered. Without `--apply` shows dry-run cluster sizes.
  - `dam faces people [--json]` — list all people with face counts.
  - `dam faces name <ID> <NAME>` — name a person.
  - `dam faces merge <TARGET> <SOURCE>` — merge two people (moves all faces from source to target).
  - `dam faces delete-person <ID>` — delete a person (faces become unassigned).
  - `dam faces unassign <FACE_ID>` — remove a face from its person.
  - `dam faces download` — download YuNet and ArcFace ONNX models.
  - Data model: `faces` table (id, asset_id, person_id, bbox, confidence, embedding, crop_path, created_at), `people` table (id, name, representative_face_id, created_at). Denormalized `face_count` column on `assets` table for fast filtering.
  - Search filters: `faces:any` / `faces:none` / `faces:N` / `faces:N+` (face count), `person:<name>` / `-person:<name>` (assigned person). Person dropdown in browse filter row.
  - Web UI: `/people` page with person gallery (cards with face crop thumbnails, names, face counts, inline rename/merge/delete, cluster button). Asset detail page faces section (face chips with crop thumbnails and confidence, "Detect faces" button, assign/unassign dropdown). Browse cards show face count badge. Batch "Detect faces" button in toolbar. API: `GET /api/asset/{id}/faces`, `POST /api/asset/{id}/detect-faces`, `POST /api/batch/detect-faces`, `GET /people`, `GET /api/people`, `PUT /api/people/{id}/name`, `POST /api/people/{id}/merge`, `DELETE /api/people/{id}`, `PUT /api/faces/{face_id}/assign`, `DELETE /api/faces/{face_id}/unassign`, `POST /api/faces/cluster`.
  - Config: `[ai] face_cluster_threshold` (default 0.5), `[ai] face_min_confidence` (default 0.5).
  - Modules: `src/face.rs` (FaceDetector: YuNet + ArcFace ONNX pipeline, multi-stride output decoder, face crop generation), `src/face_store.rs` (FaceStore: SQLite persistence, clustering, people management).

**Output formatting**:
- **Global `--json` flag**: Available on all commands. Outputs structured JSON to stdout; human-readable messages go to stderr. All data types (`SearchRow`, `AssetDetails`, `ImportResult`, `VerifyResult`, `SyncResult`, `CleanupResult`, `RelocateResult`, `DuplicateEntry`) derive `serde::Serialize`.
- **Global `--log` / `-l` flag**: Per-file progress logging for multi-file commands (import, verify, sync, refresh, cleanup, generate-previews). Each file prints `filename — status (duration)` to stderr. For `dam serve`, enables per-request logging (`METHOD /path -> STATUS (duration)` to stderr).
- **Global `--debug` / `-d` flag**: Shows stderr output from external tools (ffmpeg, dcraw, dcraw_emu) for diagnosing preview generation issues. Prints the command line and stderr to eprintln.
- **Global `--time` / `-t` flag**: Shows total elapsed time after command execution.
- **Rating**: First-class `Option<u8>` field on `Asset` (persisted in sidecar YAML and SQLite `assets.rating` column). Extracted from XMP during import. Editable via `QueryEngine::set_rating()` and web UI inline stars. Filterable in CLI search (`rating:N` exact, `rating:N+` minimum) and web UI (clickable star filter). Displayed as stars in `dam show` output and web UI browse cards/asset detail. Written back to XMP on change (see **XMP write-back** above).
- **Color label**: First-class `Option<String>` field on `Asset` (persisted in sidecar YAML and SQLite `assets.color_label` column). 7-color set: Red, Orange, Yellow, Green, Blue, Pink, Purple (CaptureOne superset of Lightroom's 5 colors). Stored as canonical title-case English names. Extracted from `xmp:Label` during import via `Asset::validate_color_label()` (case-insensitive). Editable via CLI `dam edit --label <color>` / `--clear-label`, `QueryEngine::set_color_label()`, and web UI inline color dots. Filterable in CLI search (`label:Red`) and web UI (clickable color dot filter in filter row). Displayed in `dam show` output, web UI browse cards (small colored dot), and asset detail page (interactive color dot picker with label name). Written back to XMP `xmp:Label` attribute on change (see **XMP write-back** above). Available as `{label}` placeholder in `--format` templates.
- **`search --format`**: Presets: `ids` (one UUID per line), `short` (default), `full` (with tags/description), `json` (JSON array). Custom templates: `'{id}\t{name}\t{tags}'` with placeholder substitution (`{label}` for color label). Result count suppressed when `--format` is explicit.
- **`search -q`**: Shorthand for `--format=ids`, for scripting (e.g. `for id in $(dam search -q tag:landscape); do ...`).
- **Search location health filters**: `orphan:true` (assets with zero file_location records), `missing:true` (assets where at least one location points to a non-existent file on an online volume — requires disk I/O), `stale:N` (assets with at least one location not verified in N days, or never verified), `volume:none` (assets with no locations on any currently online volume). These filters combine with all other search filters. `orphan:true` and `stale:N` are pure SQL; `missing:true` pre-computes affected asset IDs via disk checks; `volume:none` pre-computes online volume IDs from DeviceRegistry.
- **`duplicates --format`**: Same presets as search, plus `{locations}` placeholder for templates.
- **Format module** (`src/format.rs`): Template engine with `{placeholder}` substitution, escape sequences (`\t`, `\n`), preset parsing.

- **Saved searches** (smart albums): Named queries stored in `searches.toml` at the catalog root. `dam saved-search save <NAME> <QUERY> [--sort <SORT>] [--favorite]` saves a search, `dam saved-search list` lists all (favorites marked with `[*]`), `dam saved-search run <NAME> [--format <FMT>]` executes and displays results, `dam saved-search delete <NAME>` removes. Alias: `dam ss`. Supports `--json`. Each search has a `favorite: bool` field (default `false`, backward-compatible serde). Only favorites are shown as chips on the browse page. Save button on browse defaults to `favorite: true` and checks for duplicate queries before creating. A dedicated management page (`/saved-searches`) provides a table of all saved searches with favorite toggle (star), rename, and delete actions. "Manage..." link on browse page and "Searches" nav link provide access. API: `GET/POST /api/saved-searches`, `DELETE /api/saved-searches/{name}`, `PUT /api/saved-searches/{name}/favorite` (toggle), `PUT /api/saved-searches/{name}/rename`. Filter values with spaces are supported via quoting: `tag:"Fools Theater"`.
- **Collections** (static albums): Manually curated lists of asset IDs. Backed by SQLite tables (`collections`, `collection_assets`) for fast queries, plus `collections.yaml` at catalog root for persistence across `rebuild-catalog`. `dam collection create <NAME> [--description <TEXT>]`, `dam collection list`, `dam collection show <NAME> [--format <FMT>]`, `dam collection add <NAME> <ASSET_IDS...>`, `dam collection remove <NAME> <ASSET_IDS...>`, `dam collection delete <NAME>`. Alias: `dam col`. Stdin piping supported for add: `dam search -q "rating:5" | xargs dam col add "Best"`. Search filter: `collection:<name>` restricts results to collection members. Web UI: `/collections` page lists all collections with "+ New Collection" button (prompts for name and optional description). Asset detail page shows collection membership chips with × remove buttons (inline `DELETE /api/batch/collection`, chip removed from DOM on success). Browse batch toolbar has collection dropdown; context-sensitive buttons show "+ Collection" when no collection is selected in the filter dropdown and "− Collection" when a collection is active. The batch dropdown auto-selects the current collection from the filter-row dropdown. Dropdown includes a "New..." option that creates a collection inline via prompt. API: `GET/POST /api/collections`, `POST/DELETE /api/batch/collection`.
- **Stacks** (scene grouping): Lightweight anonymous groups that collapse in the browse grid to show only the "pick" image with a count badge. One stack per asset, position-ordered (index 0 = pick), auto-dissolve when ≤1 member. `dam stack create <IDS...>`, `dam stack add <REF> <IDS...>`, `dam stack remove <IDS...>`, `dam stack pick <ID>`, `dam stack dissolve <ID>`, `dam stack list`, `dam stack show <ID>`. Alias: `dam st`. Persists in `stacks.yaml`, restored during `rebuild-catalog`. Search filter: `stacked:true` / `stacked:false`. Web UI: stack badge (⊞ N) on browse cards, colored left border per stack for visual grouping, collapse/expand toggle in results bar, Stack/Unstack batch buttons, asset detail stack section with set-pick and dissolve. Data model: `stacks` SQLite table + denormalized `stack_id`/`stack_position` on `assets`. API: `POST/DELETE /api/batch/stack`, `PUT /api/asset/{id}/stack-pick`, `DELETE /api/asset/{id}/stack`.
- **Hierarchical tags**: Tags can contain `/` as hierarchy separator (e.g. `animals/birds/eagles`). Searching `tag:animals` matches all descendants. Tags page shows collapsible tree view with own-count and total-count. Interops with Lightroom `lr:hierarchicalSubject` XMP field. Internally stored with `|` separator to avoid conflicts with literal `/` in tag names.
- **Geo search filter**: `geo:any` (has GPS), `geo:none` (no GPS), `geo:<lat>,<lng>,<radius_km>` (bounding circle), `geo:<south>,<west>,<north>,<east>` (bounding box). Pure assets-table filter using denormalized `latitude`/`longitude` columns. Bounding circle converted to bounding box via `dlat = r/111, dlng = r/(111*cos(lat))`.
- **Quoted filter values**: Search query parser supports double-quoted values for multi-word filters: `tag:"Fools Theater"`, `camera:"Canon EOS R5"`, `lens:"RF 50mm f/1.2"`, `collection:"My Favorites"`, `path:"Photos/Family Trip"`. Unquoted single-word values continue to work as before.
- **Path prefix filter**: `path:<prefix>` restricts results to assets with a `file_locations.relative_path` starting with the given prefix (SQL `LIKE prefix%`). Useful for finding all assets imported from a specific directory (e.g., `path:Capture/2026-02-22`). Path values are automatically normalized in the CLI: `~` expands to `$HOME`, `./` and `../` resolve relative to the current working directory, and absolute paths matching a registered volume's mount point are stripped to volume-relative with the volume filter implicitly applied (e.g., `path:/Volumes/Photos/Capture/2026` becomes `path:Capture/2026` scoped to that volume). An explicit `volume:` filter takes precedence over the implicit one. Plain relative paths (no `./` prefix) remain volume-relative prefix matches. Composes with all other filters. Available in CLI, web UI (dedicated input in browse filter row), and saved searches.

**Configuration** (`dam.toml`):
- Parsed via `toml` crate with serde. All sections optional; missing fields use defaults. Validated on load.
- `default_volume: Option<Uuid>` — fallback volume for import.
- `[preview]`: `max_edge: u32` (default 800), `format: "jpeg"|"webp"` (default jpeg), `quality: u8` 1–100 (default 85, JPEG only; WebP is lossless via `image` crate), `smart_max_edge: u32` (default 2560), `smart_quality: u8` (default 85), `generate_on_demand: bool` (default false — when true, web server generates smart previews on first request). Stored in `PreviewConfig`. Passed to `PreviewGenerator` and `AssetService`.
- `[serve]`: `port: u16` (default 8080), `bind: String` (default "127.0.0.1"), `per_page: u32` (default 60, results per page in browse grid). CLI `--port`/`--bind`/`--per-page` flags override.
- `[import]`: `exclude: Vec<String>` (glob patterns matched against filenames via `glob_match`), `auto_tags: Vec<String>` (merged into new assets, deduplicated with XMP tags), `smart_previews: bool` (default false — when true, generates smart previews during import). Passed to `import_with_callback` and `resolve_files`.
- `[dedup]`: `prefer: Option<String>` (default path substring for `--prefer` flag). Used by CLI `dam dedup` and web UI auto-resolve as fallback when `--prefer` is not given. CLI `--prefer` overrides the config value.
- `[verify]`: `max_age_days: Option<u64>` (default max-age for incremental verify). Used by CLI `dam verify` as fallback when `--max-age` is not given. CLI `--max-age` overrides the config value.
- `[ai]`: `model: String` (default `"siglip-vit-b16-256"`, which model to use; also `siglip-vit-l16-256` available), `threshold: f32` (default 0.5, minimum confidence to suggest/apply tags), `labels: Option<String>` (path to custom labels file, one per line), `model_dir: Option<String>` (directory for ONNX model files, default `~/.cache/dam/models`), `prompt: Option<String>` (prefix for label text encoding). Feature-gated behind `--features ai`. Stored in `AiConfig`. Used by `dam auto-tag` CLI and web UI suggest/batch endpoints.
- Config structs: `CatalogConfig` in `src/config.rs` with sub-structs `PreviewConfig`, `ServeConfig`, `ImportConfig`, `DedupConfig`, `VerifyConfig`, `AiConfig`. User-facing documentation in `README.md`.
