# simple digital asset manager

## mandatory requirements

- suitable for all kinds of digital assets, especially images and videos
- all formats of RAW files must be supported (e.g. NEF, RAF etc.)
- multiple variants of the same asset must be grouped and/or navigatable (e.g. RAW / JPEG, different processing etc.)
- media files can be stored on one or more offline devices (we are talking about terrabytes)
- duplicates should be stored only once
- original files (and maybe also variants?) can move (e.g. on different storage device) transparently
- processing instructions / recipies etc. should be managed as well. This should include software like CaptureOne, Photoshop etc.

## basic technical ideas

- an original file (most of the time this is the RAW file) is never changed, so we can make it content adressable (e.g. SHA)
- metadata stored in sidecar files (how can we link media and sidecar)?
- navigation / retrieval independent of location of media files
- all should be text based
- should we use git as backend (but probably only the storage part)?

## implementation status

### implemented

- **`init`** — initialize a new catalog in the current directory (creates metadata dir, SQLite schema, volume registry, config)
- **`volume add/list/combine/remove`** — register storage volumes, list them with online/offline status, combine child volumes into a parent, or remove a volume and its data
- **`import`** — hash files (SHA-256), extract EXIF metadata, create assets/variants, write YAML sidecars, insert into SQLite catalog. `dam import <paths...> [--volume V] [--include G] [--skip G] [--dry-run]`. `--dry-run` reports what would be imported without writing to catalog, sidecar, or disk (files are still hashed for dedup detection).
  - Stem-based auto-grouping: files sharing the same filename stem in the same directory are grouped into one asset (e.g. `DSC_4521.nef` + `DSC_4521.jpg` → 1 asset, 2 variants)
  - RAW files take priority as the primary variant (defining asset identity via deterministic UUID and EXIF-based `created_at`)
  - Recipe handling: processing sidecars (`.xmp`, `.cos`, `.cot`, `.cop`, `.pp3`, `.dop`, `.on1`) are attached as Recipe records to the primary variant. Recipes are identified by location (volume + path), not content hash — re-importing after external edits updates the recipe in place and re-extracts XMP metadata. Standalone recipe imports (no co-located media) resolve to parent variants by stem + directory matching
  - Duplicate location tracking: re-importing the same content from a different path adds the new location to the existing variant
  - `--volume` overrides auto-detection of which volume the files belong to
  - `--smart` generates smart previews (2560px JPEG, configurable via `[preview] smart_max_edge`) alongside regular thumbnails during import. Can be enabled permanently via `[import] smart_previews = true` in `dam.toml`
  - `--embed` (feature-gated: `--features ai`) generates SigLIP image embeddings for visual similarity search during import. Silently skips if the AI model is not downloaded. Can be enabled permanently via `[import] embeddings = true` in `dam.toml`
  - `--add-tag <TAG>` (repeatable) adds tags to every imported asset. Merged with `[import] auto_tags` and XMP tags, deduplicated
  - `--auto-group` runs auto-grouping scoped to the "neighborhood" of imported files after normal import. Neighborhood is computed by going up one directory level from each imported file's directory to find session roots, then finding all catalog assets on the same volume under those roots. This avoids catalog-wide false positives from restarting camera counters. Combines with `--dry-run`. Skips if nothing was imported
  - **Embedded XMP extraction**: JPEG and TIFF files have their embedded XMP metadata extracted automatically during import (keywords, ratings, descriptions, labels, creator, rights). Precedence chain: EXIF (highest, direct assignment) > embedded XMP (middle) > sidecar `.xmp` (lowest). For tags, all sources merge (union). For description/rating/label, the first source that provides a value wins
  - Summary only reports non-zero stat categories
- **`search`** — search assets by text, type, tag, or format via SQLite catalog
- **`show`** — display full asset details including variants, locations, source metadata, and recipes
- **`tag`** — add or remove tags on an asset (with `--remove` flag). **XMP write-back**: tag changes are written back to `.xmp` recipe files on disk using operation-level deltas — added tags are inserted into `dc:subject`/`rdf:Bag`, removed tags are deleted. Tags added independently in CaptureOne are preserved (not overwritten). If no `dc:subject` block exists, one is injected (with `xmlns:dc` namespace if needed). After writing, the file is re-hashed and the recipe's content hash is updated. Offline volumes and missing files are silently skipped.
- **`edit`** — edit asset metadata from the CLI: `dam edit <asset-id> [--name <name>] [--description <text>] [--rating <1-5>] [--label <color>] [--date <YYYY-MM-DD>] [--clear-name] [--clear-description] [--clear-rating] [--clear-label] [--clear-date]`. At least one flag is required. Sets or clears the asset's name, description, rating, color label, and creation date. Updates both sidecar YAML and SQLite catalog. Supports `--json` for structured output.
- **`delete`** — remove assets from the catalog: `dam delete <ASSET_IDS...> [--apply] [--remove-files]`. Default is report-only mode (shows what would be deleted without making changes). `--apply` executes deletion: removes asset rows, variants, file locations, recipes, previews, smart previews, face crops, face and embedding DB records, embedding binaries, sidecar YAML files, collection memberships, and stack membership. `--remove-files` (requires `--apply`) also deletes physical media and recipe files from disk on online volumes (offline volumes are skipped with a warning). Supports stdin piping when no asset IDs are provided (`dam search -q "orphan:true" | dam delete --apply`). Supports asset ID prefix matching. Supports `--json`, `--log`, `--time` flags.
- **`export`** — copy files matching a search query to a target directory: `dam export <QUERY> <TARGET> [--layout flat|mirror] [--symlink] [--all-variants] [--include-sidecars] [--dry-run] [--overwrite]`. Default exports the best variant per asset in flat layout (filename collisions resolved by hash suffix). `--layout mirror` preserves source directory structure (multi-volume gets volume-label prefix). `--symlink` creates symlinks instead of copies. `--all-variants` exports every variant. `--include-sidecars` copies recipe files alongside. `--dry-run` reports without writing. `--overwrite` forces re-copy. Files integrity-verified via SHA-256 after copy. Supports `--json`, `--log`, `--time`.
- **`contact-sheet`** — generate a contact sheet (grid of thumbnails) as a JPEG image: `dam contact-sheet <QUERY> <OUTPUT> [--columns N] [--thumb-size N] [--title TEXT] [--metadata]`. Uses existing preview images. `--columns` sets the number of columns (default 6). `--thumb-size` sets the thumbnail size in pixels (default 200). `--title` adds a title header. `--metadata` annotates each cell with filename and rating stars. Supports `--json`, `--time`.
- **`group`** — manually group variants into one asset by content hash. Merges donor assets into the target: combines tags (union, deduplicated), moves variants, copies recipes, and re-roles donor variants from Original to Alternate (in both sidecar YAML and SQLite catalog). Donor assets, their sidecar YAML files, and catalog rows are removed after merge.
- **`split`** — split variants out of a multi-variant asset into new standalone assets: `dam split <asset-id> <variant-hashes...>`. Each specified variant becomes a new asset with its own sidecar YAML. Recipes attached to split variants move with them. The source asset retains all non-split variants. Supports `--json`, `--log`, `--time`.
- **`auto-group`** — automatically group assets by filename stem across directories: `dam auto-group [QUERY] [--apply]`. Scopes to search results (same syntax as `dam search`). Uses fuzzy prefix + separator matching: shorter stem must be a prefix of longer, and the next character must be non-alphanumeric (e.g., `Z91_8561.ARW` matches `Z91_8561-1-HighRes-(c)_2025_Thomas Herrmann.tif` but `DSC_001` does not match `DSC_0010`). Chain resolution ensures multiple export levels all resolve to the shortest root stem. Target selection prefers RAW files, then oldest by `created_at`. Dry-run by default (reports groupable stems); `--apply` executes the merge via `group()`. Supports `--json`, `--log`, `--time`. Web UI: "Group by name" button in batch toolbar calls `POST /api/batch/auto-group`
- **`rebuild-catalog`** — drop and rebuild SQLite catalog from YAML sidecar files (including recipes). Also restores collections from `collections.yaml`, stacks from `stacks.yaml`, faces from `faces.yaml`, people from `people.yaml`, ArcFace face embeddings from `embeddings/arcface/` binary files, and SigLIP image embeddings from `embeddings/<model>/` binary files
- **`migrate`** — run database schema migrations explicitly. Must be run after upgrading to a new version (all other commands check the schema version at startup via a fast O(1) query against the `schema_version` table and exit with an error if outdated). Prints the resulting schema version number. Supports `--json` (includes `schema_version` field)
- **`duplicates`** — find files with the same content hash across multiple locations, showing all volume/path pairs. `dam duplicates [--same-volume] [--cross-volume] [--volume V] [--filter-format F] [--path P] [--format FMT]`
- **`dedup`** — remove same-volume duplicate file locations: `dam dedup [--volume V] [--prefer S] [--filter-format F] [--path P] [--min-copies N] [--apply]`. Resolution heuristic prefers locations whose path contains `--prefer` substring, then most recently verified, then shortest path. Co-located recipe files are automatically cleaned up. `[dedup] prefer` in `dam.toml` provides a default. Supports `--json`, `--log`, `--time`
- **`generate-previews`** — generate missing preview thumbnails. Supports `PATHS` (resolve files on disk), `--asset`, `--volume`, `--include`/`--skip` (file type groups), and `--force` (regenerate existing). Per-file progress with `-l` flag
- **Preview generation during import** — previews are generated for each imported variant. Uses the `image` crate for standard formats (800px JPEG thumbnails), `dcraw`/`dcraw_emu` (LibRaw) for RAW files, and `ffmpeg` for videos. Non-visual formats (audio, documents, unknown) get an info card — an 800x600 JPEG showing file metadata (name, format, size, and audio properties like duration/bitrate via `lofty`). When external tools (dcraw, ffmpeg) are missing, RAW and video files also fall back to an info card. Previews stored in `previews/<hash-prefix>/<hash>.jpg`. Preview failure never blocks import.
- **`show`** now displays preview status (path if exists, "(none)" otherwise)
- **`relocate`** — copy or move all asset files (variants + recipes) to a target volume: `dam relocate <asset-id> <target-volume> [--remove-source] [--dry-run]`. Copies files with SHA-256 integrity verification, preserves relative paths, updates sidecar and catalog metadata. Without `--remove-source`, the asset gains additional locations. With `--remove-source`, source files are deleted after verified copy. `--dry-run` shows the plan without making changes.
- **`update-location`** — update a file's catalog path after it was manually moved on disk: `dam update-location <asset-id> --from <old-path> --to <new-path> [--volume <label>]`. `--to` must be an absolute path to the file's current location. `--from` can be absolute or volume-relative. Volume is auto-detected from `--to`, or specified via `--volume`. Verifies the file at `--to` has the same content hash as the catalog record. Updates both SQLite catalog and sidecar YAML. Handles both variant file locations and recipe file locations.
- **`verify`** — re-hash files on disk and compare against stored content hashes to detect corruption or bit rot: `dam verify [PATHS...] [--volume <label>] [--asset <id>] [--max-age <DAYS>] [--force]`. Without arguments, verifies all file locations on all online volumes. With paths, verifies specific files or directories. `--volume` limits to a specific volume; `--asset` limits to a specific asset. `--max-age <DAYS>` skips files verified within the given number of days (incremental verify). `--force` overrides `--max-age` and re-verifies everything. Default `max_age_days` configurable via `[verify]` in `dam.toml`. Updates `verified_at` timestamps on successful verification (persisted to both SQLite catalog and sidecar YAML for variant and recipe locations). Exits with code 1 if any mismatches are found. Modified recipe files are reported as "modified" (not "FAILED") and do not trigger exit code 1 — their stored hash is updated to reflect the new content.
- **`sync`** — reconcile catalog with disk reality after external tools move, rename, or modify files: `dam sync <PATHS...> [--volume <label>] [--apply] [--remove-stale]`. Without `--apply`, runs in report-only mode (safe default). With `--apply`, updates catalog and sidecar files for moved files and modified recipes. `--remove-stale` (requires `--apply`) removes catalog location records for confirmed-missing files. Detects: unchanged files (hash matches at expected path), moved files (known hash at new path, old path gone), new files (unknown hash), modified recipes (same path, different hash), missing files (catalog location but file gone from disk). New files are reported but not auto-imported — user runs `dam import` separately.
- **`cleanup`** — scan all file locations and recipes across online volumes, removing records for files no longer on disk, orphaned assets, and orphaned derived files: `dam cleanup [--volume <label>] [--list] [--apply]`. Without `--apply`, runs in report-only mode (safe default, predicts orphans). With `--apply`, performs seven passes: (1) removes stale location and recipe records for missing files, (2) deletes orphaned assets (all variants have zero file_locations) including their recipes, variants, faces, embeddings, previews, smart previews, face crops, embedding binaries, catalog rows, and sidecar YAML, (3) removes orphaned preview files (content hash no longer matches any variant), (4) removes orphaned smart preview files, (5) removes orphaned SigLIP embedding binaries (asset ID no longer in catalog), (6) removes orphaned face crop thumbnails (face ID no longer in catalog), (7) removes orphaned ArcFace embedding binaries. `--volume` limits stale-location scanning to a specific volume (otherwise checks all online volumes). `--list` prints only stale entries to stderr (unlike `--log` which prints all entries). Skips offline volumes with a note.
- **`sync-metadata`** — bidirectional XMP metadata sync: `dam sync-metadata [--volume <label>] [--asset <id>] [--dry-run] [--media]`. Phase 1 (Inbound): detects externally modified XMP recipe files and re-reads their metadata (keywords, rating, description, color label). Phase 2 (Outbound): writes pending DAM edits to XMP recipe files (recipes marked `pending_writeback`). Phase 3 (Media, with `--media`): re-extracts embedded XMP from JPEG/TIFF variant files. Detects conflicts when both sides changed. Reports per-recipe status as `Unchanged`, `Inbound`, `Outbound`, `Conflict`, `Missing`, or `Offline`. Supports `--json`, `--log`, `--time` flags.
- **Output formatting** — flexible output for scripting and machine consumption:
  - Global `--json` flag on all commands: outputs structured JSON to stdout, human messages to stderr
  - Global `-l` / `--log` flag: per-file progress logging for multi-file commands (import, verify, sync, cleanup, generate-previews). Prints `filename — status (duration)` to stderr. For `dam serve`, enables per-request logging (`METHOD /path -> STATUS (duration)` to stderr)
  - Global `-t` / `--time` flag: shows total elapsed time after command execution
  - Global `--debug` / `-d` flag: shows stderr output from external tools (ffmpeg, dcraw, dcraw_emu) for diagnosing preview generation issues
  - `search --format=<preset|template>`: presets are `ids` (one UUID per line), `short` (default compact), `full` (with tags/description), `json` (JSON array). Custom templates use `{placeholder}` syntax, e.g. `'{id}\t{name}\t{tags}'`. Supported placeholders: `id`, `short_id`, `name`, `filename`, `type`, `format`, `date`, `tags`, `description`, `hash`
  - `search -q` / `--quiet`: shorthand for `--format=ids`
  - Search location health filters: `orphan:true` (assets with zero file_locations), `missing:true` (at least one location points to a non-existent file), `stale:N` (at least one location not verified in N days or never verified), `volume:none` (no locations on any online volume). Combinable with all other search filters.
  - Asset ID prefix filter: `id:<prefix>` matches assets whose UUID starts with the given prefix. Useful for scripting and quick lookup in both CLI and web UI.
  - `duplicates --format=<preset|template>`: same presets, with additional `{locations}` placeholder
  - When `--format` is explicitly set, result counts are suppressed
- **`stats`** — show catalog statistics: `dam stats [--types] [--volumes] [--tags] [--verified] [--all] [--limit N]`. Without flags, shows overview (assets, variants, recipes, volumes, total size). `--types` adds asset type breakdown with percentages and top variant/recipe formats. `--volumes` adds per-volume details (asset/variant/recipe counts, size, directories, formats, verification coverage). `--tags` shows unique tag count, tagged/untagged assets, and top tags by frequency. `--verified` shows verification health (coverage, oldest/newest timestamps, per-volume breakdown). `--all` enables all sections. `--limit N` controls top-N lists (default 20). Supports `--json` for structured output.

- **`serve`** — start web UI server: `dam serve [--port <port>] [--bind <addr>] [--per-page <N>]`. Default `127.0.0.1:8080`. `--log` enables per-request logging to stderr (`METHOD /path -> STATUS (duration)`). `--per-page` sets results per page (default 60, configurable via `[serve] per_page` in `dam.toml`). Browse/search page with filter dropdowns (type, tag, volume, collection, rating), grouped multi-select format filter (formats categorized as RAW/Image/Video/Audio/Other with group-level toggles and per-format variant counts), color label filter dots, sort options, pagination, and thumbnail grid with star ratings, color label dots, and variant count badges. Browse cards show the asset's identity format (RAW format when available) rather than the preview variant's format. Multi-variant assets display a count badge (e.g. "3v"). Each asset appears once in the grid regardless of variant count (deduplicated via denormalized `best_variant_hash`). Asset detail page with preview, metadata table, inline name editing (pencil icon, text input, Save/Cancel, `PUT /api/asset/{id}/name`; empty clears name, shows filename fallback), inline tag editing (add/remove via htmx, with autocomplete suggestions), inline star rating (click to set/clear via htmx PUT), inline color label picker (7 color dots, click to set/toggle), inline description editing (pencil icon, textarea, Save/Cancel), inline date editing, rotate button (cycles 90° CW, regenerates both regular and smart previews with EXIF auto-orientation), variants table, recipes list, and collapsible source metadata. **Preview orientation**: previews are auto-oriented from EXIF tags during generation; manual rotation override persisted per asset in `preview_rotation` field. Rating filter supports minimum threshold (e.g. 3+) and exact match. Batch operations toolbar with tag (with autocomplete), rating, and color label editing. Results grid shows a loading indicator (reduced opacity) during htmx page navigation and search requests. Keyboard navigation: arrow keys navigate between browse cards (column-aware), Enter opens lightbox, Space toggles selection, Shift+Left/Right turns pages, 1–5/0 set/clear rating, Alt+1–7/0 set/clear color label, letter keys r/o/y/g/b/p/u/x for quick label — operates on focused card or batch selection. Uses axum + askama templates + htmx for partial page updates. Static assets embedded at compile time. Schema migrations run once at server startup; per-request connections use `Catalog::open_fast()`. Browse page detects htmx vs direct requests (via `HX-Request` header) to return partial or full HTML — ensures back button, reload, and bookmarks always render correctly. `/api/search` redirects non-htmx requests to `/?params`. Browse results auto-refresh on bfcache restoration (back from asset detail) to reflect changes made on the asset page. Backup status page (`/backup`) shows backup health dashboard: summary cards (total assets, at-risk count, min copies threshold), volume distribution bar chart, coverage by purpose table, volume gaps table, and at-risk link to browse page filtered to `copies:1`. **Dark mode**: toggle button (sun/moon) in nav bar switches between light and dark themes; follows OS preference via `prefers-color-scheme` when no explicit choice is stored; preference persisted in `localStorage('dam-theme')`; FOUC-preventing inline script in `<head>` applies theme before stylesheet loads; covers all pages and components (badges, stats, backup status, batch toolbar). **Grid density**: three presets — Compact (120px min, hidden metadata/stars), Normal (200px default), Large (300px min, 2-line titles) — controlled by toggle buttons (SVG grid icons) in the results bar; CSS variable `--grid-min` on `<html data-density>`; persisted in `localStorage('dam-grid-density')`; keyboard nav column count adjusts automatically via computed `gridTemplateColumns`. **Lightbox**: shared full-screen overlay component (extracted as `lightbox.html` + `lightbox_js.html` Askama includes) available on browse, detail, and stroll pages. - **Date search filters** — three query filters for filtering assets by creation date: `date:2026-02-25` (prefix match at day/month/year granularity, uses SQLite LIKE with index), `dateFrom:2026-01-15` (inclusive lower bound, string comparison on RFC 3339 dates), `dateUntil:2026-02-28` (inclusive upper bound, converted to exclusive next-day/month/year internally via `next_date_bound()`). All three compose with each other and all existing filters. Available in CLI search, web UI query input, and saved searches.
- **Calendar heatmap view** — the browse page has a Grid/Calendar view toggle (two icon buttons next to the density controls). The calendar view shows a GitHub-style year-at-a-glance heatmap with day cells colored by asset count (quartile-based 5-level scale). Navigate between years with arrow buttons and year chips. Click any day to filter the grid to that date (`date:YYYY-MM-DD` injected into search query). All existing search filters apply to the calendar aggregation. Backed by `GET /api/calendar?year=N` JSON endpoint returning `{year, counts: {"2026-02-25": 14}, years: [2024, 2025, 2026]}`. Calendar data computed by `Catalog::calendar_counts()` (reuses `build_search_where()` for filter consistency) and `Catalog::calendar_years()`. View mode persisted in `localStorage('dam-view-mode')`. Full dark mode support with CSS custom property color scales.
- **Rating** — first-class `Option<u8>` field on Asset, persisted in sidecar YAML and SQLite. Extracted from XMP `xmp:Rating` during import (conservative merge on first import, overwrite on re-import). `MicrosoftPhoto:Rating` values (percentage scale: 1, 25, 50, 75, 99) are normalized to the standard 1–5 scale via `normalize_rating()` when `xmp:Rating` is absent. Searchable via CLI `rating:N` (exact) and `rating:N+` (minimum). Displayed as stars in `dam show` output. Editable via `dam edit --rating` / `--clear-rating`, web UI inline star rating with htmx, and `PUT /api/asset/{id}/rating` endpoint (accepts `rating=N` 1-5 or `rating=0` to clear). **XMP write-back**: rating changes are automatically written back to `.xmp` recipe files on disk (string-based read-modify-write preserving all other XMP content). After writing, the file is re-hashed and the recipe's content hash is updated in catalog and sidecar. Offline volumes and missing files are silently skipped. This enables bidirectional sync with tools like CaptureOne.
- **Color label** — first-class `Option<String>` field on Asset, persisted in sidecar YAML and SQLite. 7-color set: Red, Orange, Yellow, Green, Blue, Pink, Purple (CaptureOne superset of Lightroom's 5 colors). Stored as canonical English title-case names; `None` = no label. Extracted from XMP `xmp:Label` during import (conservative merge on first import, overwrite on re-import). Searchable via CLI `label:Red` filter. Displayed in `dam show` output. Editable via `dam edit --label <color>` / `--clear-label`, web UI inline color dot picker (7 dots, click to set/toggle), and `PUT /api/asset/{id}/label` endpoint. Batch operations via `PUT /api/batch/label`. **XMP write-back**: label changes are written back to `.xmp` recipe files on disk. Clearing a label removes the `xmp:Label` attribute/element entirely (unlike rating which uses 0 as sentinel). Web UI shows color dots on browse cards, a label filter row on the browse page, and a label section in the batch toolbar.
- **`refresh`** — re-read metadata from changed sidecar/recipe files: `dam refresh [PATHS...] [--volume <label>] [--asset <id>] [--dry-run] [--media]`. Lightweight alternative to `sync` focused exclusively on recipe metadata. Iterates recipe file locations in the catalog, compares on-disk hash to stored hash, and for changed files re-extracts XMP metadata (keywords, rating, description, color label) and updates catalog + sidecar. Non-XMP recipes get hash updated without metadata extraction. `--media` additionally scans JPEG/TIFF variant files and re-extracts their embedded XMP metadata (keywords, rating, description, label, creator, rights) — useful for retroactively extracting embedded XMP from files imported before the feature existed, or after external tools edit the embedded metadata. Without arguments checks all online volumes; with paths scans recipe files under given paths; `--volume` limits to a specific volume; `--asset` limits to a specific asset. `--dry-run` reports changes without applying. Supports `--json`, `--log`, `--time` flags.
- **`saved-search`** (alias `ss`) — manage saved searches (smart albums): named queries stored in `searches.toml` at the catalog root. `dam ss save <NAME> <QUERY> [--sort <SORT>]` saves/replaces a search, `dam ss list` lists all with query and sort, `dam ss run <NAME> [--format <FMT>]` executes and displays results using the same output formats as `dam search`, `dam ss delete <NAME>` removes. Supports `--json`. Web UI: clickable chips on the browse page load saved searches into filter dropdowns. Save button captures the current search state (query text, tag, type, format, rating, label, sort). Chips show rename (…) and delete (×) buttons on hover. API: `GET/POST /api/saved-searches`, `DELETE /api/saved-searches/{name}`.
- **`collection`** (alias `col`) — manage collections (static albums): manually curated lists of asset IDs, backed by SQLite tables (`collections`, `collection_assets`) for fast queries plus `collections.yaml` at catalog root for persistence across `rebuild-catalog`. `dam col create <NAME> [--description <TEXT>]`, `dam col list`, `dam col show <NAME> [--format <FMT>]`, `dam col add <NAME> <ASSET_IDS...>`, `dam col remove <NAME> <ASSET_IDS...>`, `dam col delete <NAME>`. Stdin piping supported for `add`: `dam search -q "rating:5" | xargs dam col add "Best"`. Search filter: `collection:<name>` restricts results to collection members. Web UI: `/collections` page lists all collections with asset counts and a "+ New Collection" button (prompts for name and optional description). Asset detail page shows collection membership chips with × remove buttons. Browse filter row includes a collection dropdown (like type/format/volume) for direct collection filtering, composable with all other search criteria. Browse batch toolbar includes a collection dropdown with context-sensitive buttons: "+ Collection" when no collection is active in the filter dropdown, "− Collection" when a collection is selected (batch dropdown auto-syncs from the filter-row dropdown). Dropdown includes a "New..." option that creates a collection inline via prompt. Collections are restored from YAML during `rebuild-catalog`. API: `GET/POST /api/collections`, `POST/DELETE /api/batch/collection`.
- **Quoted filter values** — search query parser supports double-quoted values for multi-word filters: `tag:"Fools Theater"`, `camera:"Canon EOS R5"`, `lens:"RF 50mm f/1.2"`, `collection:"My Favorites"`, `path:"Photos/Family Trip"`. Unquoted single-word values continue to work as before.
- **Negation** — prefix any filter or free-text term with `-` to exclude matches: `-tag:rejected`, `-format:xmp`, `-type:other`, `-label:Red`, `-camera:phone`, `-sunset` (free-text). Negation works with all filter types. Example: `dam search "landscape -tag:processed"` finds landscapes not yet tagged "processed".
- **OR within filters (comma operator)** — use commas within a single filter value to match any of the given options (OR logic): `tag:alice,bob` (tagged alice OR bob), `format:nef,cr3` (NEF or CR3 format), `type:image,video` (images or videos), `label:Red,Orange` (Red or Orange labeled). Repeated filters combine with AND: `tag:landscape tag:sunset` requires BOTH tags. Negation and OR can be combined: `type:image,video -format:xmp` (images or videos, but not XMP files).
- **Path prefix filter** — `path:<prefix>` restricts search results to assets with a `file_locations.relative_path` starting with the given prefix (SQL `LIKE prefix%`). Useful for finding all assets imported from a specific directory or shoot folder (e.g., `path:Capture/2026-02-22`). Path values are automatically normalized in the CLI: `~` expands to `$HOME`, `./` and `../` resolve relative to the current working directory, and absolute paths matching a registered volume's mount point are stripped to volume-relative with the volume filter implicitly applied (e.g., `path:/Volumes/Photos/Capture/2026` becomes `path:Capture/2026` scoped to that volume). An explicit `volume:` filter takes precedence over the implicit one. Plain relative paths (no `./` prefix) remain volume-relative prefix matches. Composes with all other filters. Available in CLI search, web UI (dedicated text input in browse filter row), and saved searches.
- **`fix-roles`** — fix variant roles in multi-variant assets containing RAW files: `dam fix-roles [PATHS...] [--volume <label>] [--asset <id>] [--apply]`. Scans assets with 2+ variants where at least one is a RAW file. Non-RAW variants with role `Original` are re-roled to `Alternate`. Assets with only non-RAW variants are untouched (no clear master to determine roles). Dry-run by default (reports what would change); `--apply` writes changes to both sidecar YAML and SQLite catalog. Supports `--json`, `--log`, `--time`. Useful for correcting roles on assets imported before auto-grouping role assignment was fixed.
- **Configuration** (`dam.toml`) — all settings live in `dam.toml` at the catalog root (created by `dam init`). Every section and field is optional; missing values use defaults. CLI flags override the corresponding `dam.toml` values. Top-level `default_volume: Option<Uuid>` sets the fallback volume for import. Sections:
  - `[preview]`: `max_edge` (default 800), `format` ("jpeg"/"webp", default jpeg), `quality` (1–100, default 85), `smart_max_edge` (default 2560), `smart_quality` (default 85), `generate_on_demand` (default false)
  - `[serve]`: `port` (default 8080), `bind` (default "127.0.0.1"), `per_page` (default 60), `stroll_neighbors` (default 12), `stroll_neighbors_max` (default 25), `stroll_fanout` (default 5), `stroll_fanout_max` (default 10), `stroll_discover_pool` (default 80)
  - `[import]`: `exclude` (glob patterns), `auto_tags` (merged into new assets), `smart_previews` (default false), `embeddings` (default false, feature-gated)
  - `[dedup]`: `prefer` (default path substring for duplicate resolution)
  - `[verify]`: `max_age_days` (default max-age for incremental verify)
  - `[ai]`: `model` (default "siglip-vit-b16-256"), `threshold` (default 0.5), `labels` (custom labels file path), `model_dir` (default `~/.cache/dam/models`), `prompt` (label text encoding prefix), `execution_provider` ("auto"/"cpu"/"coreml", default "auto"), `face_cluster_threshold` (default 0.5), `face_min_confidence` (default 0.5). Feature-gated behind `--features ai`
  - `[vlm]`: `endpoint` (default "http://localhost:11434"), `model` (default "qwen2.5vl:3b"), `max_tokens` (default 200), `prompt` (custom override), `timeout` (default 120s), `temperature` (default 0.7), `mode` ("describe"/"tags"/"both", default "describe"), `concurrency` (default 1, reserved). No feature gate
  - `[contact_sheet]`: `layout` (dense/standard/large), `paper` (a4/letter/a3), `landscape`, `title`, `fields`, `sort`, `group_by`, `label_style`, `copyright`, `margin`, `quality`
  - Config structs: `CatalogConfig` in `src/config.rs` with sub-structs `PreviewConfig`, `ServeConfig`, `ImportConfig`, `DedupConfig`, `VerifyConfig`, `AiConfig`, `VlmConfig`
- **`stack`** (alias `st`) — manage stacks (scene grouping): lightweight anonymous groups of assets that collapse in the browse grid to show only the "pick" image with a count badge. One stack per asset, position-ordered (index 0 = pick), auto-dissolve when ≤1 member remains. `dam stack create <IDS...>` creates a stack, `dam stack add <REF> <IDS...>` adds to an existing stack, `dam stack remove <IDS...>` removes from stacks, `dam stack pick <ID>` sets the stack pick, `dam stack dissolve <ID>` dissolves the entire stack, `dam stack list` lists all stacks, `dam stack show <ID>` shows stack members. Stacks persist in `stacks.yaml` at the catalog root and are restored during `rebuild-catalog`. Search filter: `stacked:true` / `stacked:false`. Web UI: stack badge on browse cards, colored left border per stack for visual grouping, collapse/expand toggle in results bar, Stack/Unstack batch buttons, asset detail stack members section with set-pick and dissolve actions. Calendar heatmap respects collapse state. All subcommands support `--json`.
- **Hierarchical tags** — tags can contain `/` as a hierarchy separator (e.g. `animals/birds/eagles`). Searching `tag:animals` matches all descendants. The tags page shows a collapsible tree view with own-count and total-count columns. Interoperates with Lightroom's `lr:hierarchicalSubject` XMP field: hierarchical subjects are imported, merged with flat `dc:subject` tags (components deduplicated), and written back on change. Internally stored with `|` separator to avoid conflicts with literal `/` in tag names.
- **Faceted browse sidebar** — toggleable read-only sidebar on the browse page showing statistical distribution of current results by rating (with bar chart), color label (with color dots), format, volume, tags (top 30), year (with bar chart), and geotagged asset count. Each section is collapsible with state persisted in the browser. Toggle via results bar button or `f` keyboard shortcut. Sidebar preference persisted in localStorage. Hidden on viewports <768px. Backed by `GET /api/facets` endpoint running 8 aggregate queries reusing `build_search_where()` for filter consistency. Counts update automatically on every search change.
- **`writeback`** — write pending metadata changes to XMP recipe files: `dam writeback [--volume <label>] [--asset <id>] [--all] [--dry-run]`. Without flags, processes only recipes marked `pending_writeback=1` (i.e., recipes whose volume was offline when a metadata change was made). `--all` writes current metadata to all XMP recipes regardless of the pending flag. For each recipe: reads current asset metadata (rating, label, tags, description), writes all four fields to the XMP file, re-hashes, updates recipe content hash, clears the pending flag. Supports `--json` (`WritebackResult` with `written`, `skipped`, `failed`, `errors`, `dry_run`), `--log`, `--time`.
- **`describe`** — generate natural language image descriptions via local vision-language models (Ollama or OpenAI-compatible API): `dam describe [--query <Q>] [--asset <id>] [--volume <label>] [--mode describe|tags|both] [--apply] [--force] [--dry-run]`. Sends the best available preview image to a VLM endpoint and receives a text description, tag suggestions, or both. `--mode describe` (default) generates a prose description; `--mode tags` generates keyword tags; `--mode both` makes two separate calls. `--apply` writes the results to asset metadata (description and/or tags); without it, results are printed only. `--force` overwrites existing descriptions. `--dry-run` shows what would be sent without calling the VLM. Uses OpenAI-compatible API format with Ollama native API fallback. Configurable via `[vlm]` section in `dam.toml` (endpoint, model, max_tokens, prompt, timeout, temperature, mode). No feature gate — works with any build via HTTP calls to external VLM server. VLM health check at startup with 5s timeout. Truncated JSON recovery for malformed VLM responses. Tag deduplication against existing asset tags. Web UI: "Describe" button on asset detail page and batch "Describe" in toolbar. `vlm_enabled` bool controls button visibility. Supports `--json`, `--log`, `--time`.
- **`backup-status`** — show which assets have insufficient backup copies. Reports asset count, at-risk count (only on a single volume), min copies threshold, per-volume distribution, and cross-volume coverage. Web UI: `/backup` page (see serve command). Supports `--json`.
- **`fix-dates`** — fix missing or incorrect asset creation dates by re-extracting from EXIF metadata: `dam fix-dates [PATHS...] [--volume <label>] [--asset <id>] [--apply]`. Scans assets where `created_at` may be wrong (e.g., imported before EXIF extraction was improved). Re-reads EXIF `DateTimeOriginal` from variant files on disk. Dry-run by default. Supports `--json`, `--log`, `--time`.
- **`fix-recipes`** — fix recipe attachment issues: `dam fix-recipes [PATHS...] [--volume <label>] [--asset <id>] [--apply]`. Re-scans recipe files and ensures they are correctly attached to their parent variants. Dry-run by default. Supports `--json`, `--log`, `--time`.
- **Smart previews** — a second preview tier at 2560px (configurable via `[preview] smart_max_edge`). Stored in `smart_previews/<hash-prefix>/<hash>.jpg`. Generated via `dam import --smart`, `[import] smart_previews = true` in `dam.toml`, detail page "Generate smart preview" button (`POST /api/asset/{id}/smart-preview`), or on-demand (`[preview] generate_on_demand = true` — web server generates on first request). Web UI shows regular preview instantly, background-loads smart preview with pulsing "HD" badge, swaps on success. Smart previews enable zoom and pan in lightbox, detail, and compare views (mouse wheel/drag/keyboard `,` `.` `+` `-`). `generate-previews --upgrade` regenerates previews for assets where a better variant (export/processed) now exists.
- **Export-based preview selection** — display logic prefers Export > Processed > Original > Alternate variant previews (skips Sidecar role). Within the same role, standard image formats (jpg/jpeg/png/tiff/tif/webp) are preferred over RAW, with file size as tiebreaker. Implemented via `best_preview_index()` / `best_preview_index_details()` in `src/models/variant.rs`. Used by `dam show`, web UI asset detail page, and `generate-previews` catalog mode.
- **Preview rotation** — `preview_rotation` field on Asset (persisted in sidecar YAML + SQLite) applies additional rotation on top of EXIF auto-orientation. Web UI "Rotate" button on detail page cycles 90° CW (0→90→180→270→0), regenerating both regular and smart previews. Preview HTTP responses include `Cache-Control: no-cache` so browsers revalidate after rotation.
- **Map view** — Grid/Calendar/Map view toggle on the browse page. Map shows geotagged assets as clustered markers on OpenStreetMap using Leaflet.js + MarkerCluster (embedded as static assets). GPS coordinates parsed to decimal degrees during import, denormalized as `latitude`/`longitude` columns on the `assets` table with composite index, backfilled on migration. `geo:` search filter: `geo:any` (has GPS), `geo:none` (no GPS), `geo:<lat>,<lng>,<radius_km>` (bounding circle), `geo:<south>,<west>,<north>,<east>` (bounding box). Bounding circle converted to bounding box via `dlat = r/111, dlng = r/(111*cos(lat))`. `GET /api/map` endpoint reuses `build_search_where()` for full filter consistency. Thumbnail popups open the navigable lightbox; metadata links to detail page. Dark mode tile inversion via CSS filter. `m` keyboard shortcut. Standalone lightbox mode via `openWithData()` for map markers not on current grid page.
- **Compare view** — `/compare?ids=id1,id2,...` side-by-side comparison of 2–4 assets. Flex columns with preview images using CSS transform zoom/pan. Sync zoom toggle broadcasts transforms across all columns. Interactive rating/label per column. EXIF display. Keyboard: arrows for column focus, `d` detail page, `s` sync toggle, `,` `.` `+` `-` zoom, `0`–`5` rating, Alt+1–7 label. "Compare" button in batch toolbar (requires 2–4 selected). Smart preview loading with HD badge.

- **Stroll page** (feature-gated: `--features ai`) — `/stroll` visual similarity exploration page. Radial graph layout with a center asset and 5–25 satellite neighbors (default 12) arranged in a circle, connected by SVG lines. Click or press Enter on a satellite to make it the new center and load new neighbors. Arrow keys cycle focus through satellites (Left/Right rotate, Enter navigates). `d` opens the detail page for the focused asset. Keyboard shortcuts for rating (0–5) and color label (r/o/y/g/b/p/u/x, Alt+0–7) work on the focused asset. `s` key opens the "Stroll from here" action from browse/detail pages. Collapsible filter bar (shared partial with browse page, toggled via Shift+F) restricts neighbor queries to the filter subset. Neighbor count configurable via UI slider (5–25). Entry points: nav bar "Stroll" link, detail page "Stroll from here" button, `/stroll?id=<asset-id>` URL (without `id`, picks a random asset with an embedding). Browser history via `pushState` for back-button navigation. Backed by `GET /api/stroll/neighbors?id=<asset-id>&limit=12&q=<optional-query>` endpoint returning center metadata and neighbor list with preview URLs and similarity scores. Dark mode support. Responsive: reduced satellite count and radius on narrow viewports. **Fan-Out slider** (0–10, default configurable via `[serve] stroll_fanout`) in the bottom-left control panel: on satellite focus, fetches that satellite's neighbors, filters out already-visible assets, and renders L2 (transitive) neighbors as smaller thumbnails in a direction-dependent radius behind the satellite. Focused satellite pulls 30% toward center when fan-out is active. L2 results cached per satellite, cleared on navigation. Clicking an L2 thumbnail navigates to it. **Stroll modes**: three mode selector buttons — Nearest (pure similarity ranking), Discover (randomized selection from a wider candidate pool for variety, pool size configurable via `[serve] stroll_discover_pool`), Explore (maximally diverse selection that avoids similarity to already-visible assets). **"Other shoots" toggle**: excludes assets sharing the same directory as the center image, surfacing visually similar images from different shoots/sessions.
- **Collapsible filter bar** — the search filter bar is extracted into a shared partial template used by both the browse page and the stroll page. On the stroll page, the filter bar is collapsed by default and toggled via Shift+F. On the browse page, the filter bar remains always visible by default. The Shift+F keyboard shortcut toggles the filter bar on both pages.
- **`similar:` search filter** (feature-gated: `--features ai`) — `similar:<asset-id>` or `similar:<asset-id>:<limit>` finds visually similar assets using stored SigLIP embeddings. Returns the top N most similar assets (default 20). Requires embeddings to have been generated via `dam embed` or `dam import --embed`. The reference asset ID supports prefix matching. Composes with all other filters (e.g. `similar:72a0bb4b rating:3+ tag:landscape`). Available in CLI search, web UI query input, and saved searches.
- **`embed:` search filter** — `embed:any` (assets with stored embeddings), `embed:none` (assets without embeddings). Pure SQL EXISTS subqueries on the `embeddings` table. Available in CLI and web UI.

- **`auto-tag`** (feature-gated: `--features ai`) — AI-powered zero-shot image classification using SigLIP vision-language models via ONNX Runtime. `dam auto-tag [--query <QUERY>] [--asset <id>] [--volume <label>] [--model <id>] [--threshold 0.1] [--labels <file>] [--apply] [--download] [--remove-model] [--list-models] [--similar <id>]`. Requires at least one scope filter (`--query`, `--asset`, `--volume`, or `--similar`) to prevent accidental full-catalog processing. Classifies images against a configurable tag vocabulary (~100 default photography categories). Each image is encoded into an embedding, scored against label embeddings using sigmoid scoring, and labels above the threshold are suggested. Report-only by default; `--apply` writes tags. Two models available: ViT-B/16-256 (768-dim, ~207 MB, default) and ViT-L/16-256 (1024-dim, ~670 MB). Select with `--model` or `[ai] model` in `dam.toml`. Model files downloaded from HuggingFace on first use. Model management: `--download`, `--remove-model`, `--list-models`. Visual similarity search: `--similar <id>` finds 20 most similar assets by cosine similarity on stored embeddings (model-scoped). SQLite table: `embeddings(asset_id TEXT, embedding BLOB, model TEXT, PRIMARY KEY (asset_id, model))`. Modules: `ai.rs` (model wrapper with `ModelSpec` registry), `model_manager.rs` (download/cache), `embedding_store.rs` (vector storage + in-memory `EmbeddingIndex` for fast similarity search). Configurable via `[ai]` section in `dam.toml` (model, threshold, labels, model_dir, prompt template). Supports `--json`, `--log`, `--time`.
- **`embed`** (feature-gated: `--features ai`) — batch-generate image embeddings for visual similarity search without tagging. `dam embed [--query <Q>] [--asset <id>] [--volume <label>] [--model <id>] [--force] [--export]`. Requires at least one scope filter (unless `--export`). For each matching asset: finds the best available image (smart preview > regular preview > original on online volume), encodes it with SigLIP, and stores the embedding in the `embeddings` table and as a binary file under `embeddings/<model>/`. Skips assets that already have an embedding for the active model (override with `--force`). `--export` dumps all existing embeddings from SQLite to binary files (no scope required). Reports embedded/skipped/error counts. Supports `--json`, `--log`, `--time`. Web UI "Suggest tags" and batch "Auto-tag" also store embeddings opportunistically.
- **In-memory embedding index** — `EmbeddingIndex` struct in `embedding_store.rs` loads all embeddings for a model into a contiguous `Vec<f32>` buffer on first similarity query, then caches in server memory. Similarity search uses dot product (embeddings are L2-normalized by SigLIP) with a min-heap for top-K selection. At 100k assets (~300 MB), search completes in <10ms. The index is updated in-place when new embeddings are stored via any code path (CLI, web suggest, web batch, web find-similar).
- **`faces`** (feature-gated: `--features ai`) — face detection, recognition, and people management using YuNet (detection) and ArcFace (recognition) ONNX models.
  - `dam faces detect [--query <Q>] [--asset <id>] [--volume <label>] [--apply]` — detect faces in images, store bounding boxes, 512-dim embeddings, and generate 150×150 JPEG crop thumbnails in `faces/`. Reports faces found per asset. Supports `--json`, `--log`, `--time`.
  - `dam faces cluster [--query <Q>] [--asset <id>] [--volume <label>] [--threshold <F>] [--apply]` — group similar face embeddings into unnamed person groups using greedy single-linkage clustering. Default threshold 0.5 (configurable via `[ai] face_cluster_threshold`). Scope filters limit which faces are clustered.
  - `dam faces people [--json]` — list all people with face counts.
  - `dam faces name <ID> <NAME>` — name a person.
  - `dam faces merge <TARGET> <SOURCE>` — merge two people.
  - `dam faces delete-person <ID>` — delete a person.
  - `dam faces unassign <FACE_ID>` — remove a face from its person.
  - `dam faces download` — download YuNet and ArcFace ONNX models.
  - `dam faces export` — export faces and people from SQLite to `faces.yaml`/`people.yaml` and ArcFace embeddings to binary files. One-time migration for existing data.
  - Search filters: `faces:any` / `faces:none` / `faces:N` / `faces:N+` (face count), `person:<name>` (assigned person).
  - Web UI: `/people` page with person gallery, asset detail face chips, browse face filters, batch face detection.
  - Dual persistence: face records, people, and ArcFace embeddings are stored in both SQLite and files (YAML + binary) for rebuild resilience. All write paths (CLI detect/cluster/name/merge/delete, web UI detect/assign/unassign/create/name/merge/delete/cluster) persist to both stores.
