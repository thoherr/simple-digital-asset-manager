# simple digital asset manager

## mandatory requirements

- suitable for all kinds of digital assets, especially images and videos
- all formats of RAW files must be supported (e.g. NEF, RAF etc.)
- multiple variants of the same asset must be grouped and/or navigatable (e.g. RAW / JPEG, different processing etc.)
- media files can be stored on one or more offline devices (we are talking about terrabytes)
- duplicates should be stored only once
- original files (and maybe also variants?) can move (e.g. on different storage device) transparently
- processing instructions / recipies etc. should be managed as well. This should include software like CaptureOne, Photoshop etc.

## basic technical ideas

- an original file (most of the time this is the RAW file) is never changed, so we can make it content adressable (e.g. SHA)
- metadata stored in sidecar files (how can we link media and sidecar)?
- navigation / retrieval independent of location of media files
- all should be text based
- should we use git as backend (but probably only the storage part)?

## implementation status

### implemented

- **`init`** — initialize a new catalog in the current directory (creates metadata dir, SQLite schema, volume registry, config)
- **`volume add/list/combine/remove`** — register storage volumes, list them with online/offline status, combine child volumes into a parent, or remove a volume and its data
- **`import`** — hash files (SHA-256), extract EXIF metadata, create assets/variants, write YAML sidecars, insert into SQLite catalog. `dam import <paths...> [--volume V] [--include G] [--skip G] [--dry-run]`. `--dry-run` reports what would be imported without writing to catalog, sidecar, or disk (files are still hashed for dedup detection).
  - Stem-based auto-grouping: files sharing the same filename stem in the same directory are grouped into one asset (e.g. `DSC_4521.nef` + `DSC_4521.jpg` → 1 asset, 2 variants)
  - RAW files take priority as the primary variant (defining asset identity via deterministic UUID and EXIF-based `created_at`)
  - Recipe handling: processing sidecars (`.xmp`, `.cos`, `.cot`, `.cop`, `.pp3`, `.dop`, `.on1`) are attached as Recipe records to the primary variant. Recipes are identified by location (volume + path), not content hash — re-importing after external edits updates the recipe in place and re-extracts XMP metadata. Standalone recipe imports (no co-located media) resolve to parent variants by stem + directory matching
  - Duplicate location tracking: re-importing the same content from a different path adds the new location to the existing variant
  - `--volume` overrides auto-detection of which volume the files belong to
  - Summary only reports non-zero stat categories
- **`search`** — search assets by text, type, tag, or format via SQLite catalog
- **`show`** — display full asset details including variants, locations, source metadata, and recipes
- **`tag`** — add or remove tags on an asset (with `--remove` flag). **XMP write-back**: tag changes are written back to `.xmp` recipe files on disk using operation-level deltas — added tags are inserted into `dc:subject`/`rdf:Bag`, removed tags are deleted. Tags added independently in CaptureOne are preserved (not overwritten). If no `dc:subject` block exists, one is injected (with `xmlns:dc` namespace if needed). After writing, the file is re-hashed and the recipe's content hash is updated. Offline volumes and missing files are silently skipped.
- **`edit`** — edit asset metadata from the CLI: `dam edit <asset-id> [--name <name>] [--description <text>] [--rating <1-5>] [--label <color>] [--date <YYYY-MM-DD>] [--clear-name] [--clear-description] [--clear-rating] [--clear-label] [--clear-date]`. At least one flag is required. Sets or clears the asset's name, description, rating, color label, and creation date. Updates both sidecar YAML and SQLite catalog. Supports `--json` for structured output.
- **`group`** — manually group variants into one asset by content hash. Merges donor assets into the target: combines tags (union, deduplicated), moves variants, copies recipes, and re-roles donor variants from Original to Export (in both sidecar YAML and SQLite catalog). Donor assets, their sidecar YAML files, and catalog rows are removed after merge.
- **`auto-group`** — automatically group assets by filename stem across directories: `dam auto-group [QUERY] [--apply]`. Scopes to search results (same syntax as `dam search`). Uses fuzzy prefix + separator matching: shorter stem must be a prefix of longer, and the next character must be non-alphanumeric (e.g., `Z91_8561.ARW` matches `Z91_8561-1-HighRes-(c)_2025_Thomas Herrmann.tif` but `DSC_001` does not match `DSC_0010`). Chain resolution ensures multiple export levels all resolve to the shortest root stem. Target selection prefers RAW files, then oldest by `created_at`. Dry-run by default (reports groupable stems); `--apply` executes the merge via `group()`. Supports `--json`, `--log`, `--time`. Web UI: "Group by name" button in batch toolbar calls `POST /api/batch/auto-group`
- **`rebuild-catalog`** — drop and rebuild SQLite catalog from YAML sidecar files (including recipes)
- **`duplicates`** — find files with the same content hash across multiple locations, showing all volume/path pairs. `dam duplicates [--same-volume] [--cross-volume] [--volume V] [--filter-format F] [--path P] [--format FMT]`
- **`dedup`** — remove same-volume duplicate file locations: `dam dedup [--volume V] [--prefer S] [--filter-format F] [--path P] [--min-copies N] [--apply]`. Resolution heuristic prefers locations whose path contains `--prefer` substring, then most recently verified, then shortest path. Co-located recipe files are automatically cleaned up. `[dedup] prefer` in `dam.toml` provides a default. Supports `--json`, `--log`, `--time`
- **`generate-previews`** — generate missing preview thumbnails. Supports `PATHS` (resolve files on disk), `--asset`, `--volume`, `--include`/`--skip` (file type groups), and `--force` (regenerate existing). Per-file progress with `-l` flag
- **Preview generation during import** — previews are generated for each imported variant. Uses the `image` crate for standard formats (800px JPEG thumbnails), `dcraw`/`dcraw_emu` (LibRaw) for RAW files, and `ffmpeg` for videos. Non-visual formats (audio, documents, unknown) get an info card — an 800x600 JPEG showing file metadata (name, format, size, and audio properties like duration/bitrate via `lofty`). When external tools (dcraw, ffmpeg) are missing, RAW and video files also fall back to an info card. Previews stored in `previews/<hash-prefix>/<hash>.jpg`. Preview failure never blocks import.
- **`show`** now displays preview status (path if exists, "(none)" otherwise)
- **`relocate`** — copy or move all asset files (variants + recipes) to a target volume: `dam relocate <asset-id> <target-volume> [--remove-source] [--dry-run]`. Copies files with SHA-256 integrity verification, preserves relative paths, updates sidecar and catalog metadata. Without `--remove-source`, the asset gains additional locations. With `--remove-source`, source files are deleted after verified copy. `--dry-run` shows the plan without making changes.
- **`update-location`** — update a file's catalog path after it was manually moved on disk: `dam update-location <asset-id> --from <old-path> --to <new-path> [--volume <label>]`. `--to` must be an absolute path to the file's current location. `--from` can be absolute or volume-relative. Volume is auto-detected from `--to`, or specified via `--volume`. Verifies the file at `--to` has the same content hash as the catalog record. Updates both SQLite catalog and sidecar YAML. Handles both variant file locations and recipe file locations.
- **`verify`** — re-hash files on disk and compare against stored content hashes to detect corruption or bit rot: `dam verify [PATHS...] [--volume <label>] [--asset <id>]`. Without arguments, verifies all file locations on all online volumes. With paths, verifies specific files or directories. `--volume` limits to a specific volume; `--asset` limits to a specific asset. Updates `verified_at` timestamps on successful verification. Exits with code 1 if any mismatches are found. Modified recipe files are reported as "modified" (not "FAILED") and do not trigger exit code 1 — their stored hash is updated to reflect the new content.
- **`sync`** — reconcile catalog with disk reality after external tools move, rename, or modify files: `dam sync <PATHS...> [--volume <label>] [--apply] [--remove-stale]`. Without `--apply`, runs in report-only mode (safe default). With `--apply`, updates catalog and sidecar files for moved files and modified recipes. `--remove-stale` (requires `--apply`) removes catalog location records for confirmed-missing files. Detects: unchanged files (hash matches at expected path), moved files (known hash at new path, old path gone), new files (unknown hash), modified recipes (same path, different hash), missing files (catalog location but file gone from disk). New files are reported but not auto-imported — user runs `dam import` separately.
- **`cleanup`** — scan all file locations and recipes across online volumes, removing records for files no longer on disk, orphaned assets, and orphaned previews: `dam cleanup [--volume <label>] [--list] [--apply]`. Without `--apply`, runs in report-only mode (safe default, predicts orphans). With `--apply`, performs three passes: (1) removes stale location and recipe records for missing files, (2) deletes orphaned assets (all variants have zero file_locations) including their recipes, variants, catalog rows, and sidecar YAML, (3) removes orphaned preview files (content hash no longer matches any variant). `--volume` limits stale-location scanning to a specific volume (otherwise checks all online volumes). `--list` prints only stale entries to stderr (unlike `--log` which prints all entries). Skips offline volumes with a note.
- **Output formatting** — flexible output for scripting and machine consumption:
  - Global `--json` flag on all commands: outputs structured JSON to stdout, human messages to stderr
  - Global `-l` / `--log` flag: per-file progress logging for multi-file commands (import, verify, sync, cleanup, generate-previews). Prints `filename — status (duration)` to stderr. For `dam serve`, enables per-request logging (`METHOD /path -> STATUS (duration)` to stderr)
  - Global `-t` / `--time` flag: shows total elapsed time after command execution
  - Global `--debug` / `-d` flag: shows stderr output from external tools (ffmpeg, dcraw, dcraw_emu) for diagnosing preview generation issues
  - `search --format=<preset|template>`: presets are `ids` (one UUID per line), `short` (default compact), `full` (with tags/description), `json` (JSON array). Custom templates use `{placeholder}` syntax, e.g. `'{id}\t{name}\t{tags}'`. Supported placeholders: `id`, `short_id`, `name`, `filename`, `type`, `format`, `date`, `tags`, `description`, `hash`
  - `search -q` / `--quiet`: shorthand for `--format=ids`
  - Search location health filters: `orphan:true` (assets with zero file_locations), `missing:true` (at least one location points to a non-existent file), `stale:N` (at least one location not verified in N days or never verified), `volume:none` (no locations on any online volume). Combinable with all other search filters.
  - `duplicates --format=<preset|template>`: same presets, with additional `{locations}` placeholder
  - When `--format` is explicitly set, result counts are suppressed
- **`stats`** — show catalog statistics: `dam stats [--types] [--volumes] [--tags] [--verified] [--all] [--limit N]`. Without flags, shows overview (assets, variants, recipes, volumes, total size). `--types` adds asset type breakdown with percentages and top variant/recipe formats. `--volumes` adds per-volume details (asset/variant/recipe counts, size, directories, formats, verification coverage). `--tags` shows unique tag count, tagged/untagged assets, and top tags by frequency. `--verified` shows verification health (coverage, oldest/newest timestamps, per-volume breakdown). `--all` enables all sections. `--limit N` controls top-N lists (default 20). Supports `--json` for structured output.

- **`serve`** — start web UI server: `dam serve [--port <port>] [--bind <addr>]`. Default `127.0.0.1:8080`. `--log` enables per-request logging to stderr (`METHOD /path -> STATUS (duration)`). Browse/search page with filter dropdowns (type, tag, format, volume, collection, rating), color label filter dots, sort options, pagination (60 per page), and thumbnail grid with star ratings, color label dots, and variant count badges. Browse cards show the asset's identity format (RAW format when available) rather than the preview variant's format. Multi-variant assets display a count badge (e.g. "3v"). Each asset appears once in the grid regardless of variant count (deduplicated via denormalized `best_variant_hash`). Asset detail page with preview, metadata table, inline name editing (pencil icon, text input, Save/Cancel, `PUT /api/asset/{id}/name`; empty clears name, shows filename fallback), inline tag editing (add/remove via htmx), inline star rating (click to set/clear via htmx PUT), inline color label picker (7 color dots, click to set/toggle), inline description editing (pencil icon, textarea, Save/Cancel), variants table, recipes list, and collapsible source metadata. Rating filter supports minimum threshold (e.g. 3+) and exact match. Batch operations toolbar with tag, rating, and color label editing. Keyboard navigation: arrow keys navigate between browse cards (column-aware), Enter opens detail page, Space toggles selection, 1–5/0 set/clear rating, Alt+1–7/0 set/clear color label, letter keys r/o/y/g/b/p/u/x for quick label — operates on focused card or batch selection. Uses axum + askama templates + htmx for partial page updates. Static assets embedded at compile time. Schema migrations run once at server startup; per-request connections use `Catalog::open_fast()`. Browse page detects htmx vs direct requests (via `HX-Request` header) to return partial or full HTML — ensures back button, reload, and bookmarks always render correctly. `/api/search` redirects non-htmx requests to `/?params`. Browse results auto-refresh on bfcache restoration (back from asset detail) to reflect changes made on the asset page. Backup status page (`/backup`) shows backup health dashboard: summary cards (total assets, at-risk count, min copies threshold), volume distribution bar chart, coverage by purpose table, volume gaps table, and at-risk link to browse page filtered to `copies:1`. **Dark mode**: toggle button (sun/moon) in nav bar switches between light and dark themes; follows OS preference via `prefers-color-scheme` when no explicit choice is stored; preference persisted in `localStorage('dam-theme')`; FOUC-preventing inline script in `<head>` applies theme before stylesheet loads; covers all pages and components (badges, stats, backup status, batch toolbar). **Grid density**: three presets — Compact (120px min, hidden metadata/stars), Normal (200px default), Large (300px min, 2-line titles) — controlled by toggle buttons (SVG grid icons) in the results bar; CSS variable `--grid-min` on `<html data-density>`; persisted in `localStorage('dam-grid-density')`; keyboard nav column count adjusts automatically via computed `gridTemplateColumns`. **Lightbox**: clicking a browse card thumbnail opens a full-screen overlay instead of navigating to the asset detail page; top bar shows counter (e.g. "3 / 156"), asset name, info panel toggle, detail page link, and close button; prev/next navigation via side-panel arrow buttons (flex layout, full-height click targets); side info panel (320px, toggled with `i` key or toolbar button) shows metadata table, interactive rating stars, and interactive color label dots with API write-back; rating/label changes update the card's data attributes and refresh the grid behind; keyboard shortcuts (Escape close, Left/Right navigate, i toggle info, 0-5 rating, Alt+0-7/r/o/y/g/b/p/u/x label) captured in bubble phase with `stopImmediatePropagation` to prevent browse keyboard nav from firing; lightbox state exposed via `window.damLightbox`; browse keyboard nav IIFE guards with `damLightbox.isOpen()`; Enter on focused card opens lightbox instead of detail page. Top bar also displays interactive rating stars and color label dots (always visible without opening info panel); `d` key opens detail page for current asset. **Keyboard help panel**: `?` key (or nav bar button) opens a global overlay showing all keyboard shortcuts for the current page; registered per-page via `window.damHelp.registerShortcuts()`; all other keyboard handlers guard with `damHelp.isOpen()`. **Detail page navigation**: Prev/Next buttons and Left/Right arrow keys navigate between adjacent browse results; browse page stores card IDs in `sessionStorage('dam-browse-ids')` for unlimited multi-hop; Escape and Back return to stored browse URL; `l` key returns to lightbox; rating (0-5) and label (Alt/Option+0-7, r/o/y/g/b/p/u/x) keyboard shortcuts via htmx; Alt/Option+number uses `e.code` for macOS compatibility. **Duplicates page** (`/duplicates`): shows duplicate file groups with summary cards (total groups, wasted space, same-volume count), mode tabs (All / Same Volume / Cross Volume), filters (path prefix, format, volume), and per-location "Remove" buttons that delete file copies from disk. "Auto-resolve" button removes all same-volume duplicates using the `dam dedup` heuristic. Each group header shows a clickable preview thumbnail; clicking opens a lightbox overlay with prev/next navigation (Left/Right arrow keys), `d` key to open asset detail page, Escape to close. Detail page back navigation returns to the duplicates page. API: `POST /api/dedup/resolve`, `DELETE /api/dedup/location`.
- **Date search filters** — three query filters for filtering assets by creation date: `date:2026-02-25` (prefix match at day/month/year granularity, uses SQLite LIKE with index), `dateFrom:2026-01-15` (inclusive lower bound, string comparison on RFC 3339 dates), `dateUntil:2026-02-28` (inclusive upper bound, converted to exclusive next-day/month/year internally via `next_date_bound()`). All three compose with each other and all existing filters. Available in CLI search, web UI query input, and saved searches.
- **Calendar heatmap view** — the browse page has a Grid/Calendar view toggle (two icon buttons next to the density controls). The calendar view shows a GitHub-style year-at-a-glance heatmap with day cells colored by asset count (quartile-based 5-level scale). Navigate between years with arrow buttons and year chips. Click any day to filter the grid to that date (`date:YYYY-MM-DD` injected into search query). All existing search filters apply to the calendar aggregation. Backed by `GET /api/calendar?year=N` JSON endpoint returning `{year, counts: {"2026-02-25": 14}, years: [2024, 2025, 2026]}`. Calendar data computed by `Catalog::calendar_counts()` (reuses `build_search_where()` for filter consistency) and `Catalog::calendar_years()`. View mode persisted in `localStorage('dam-view-mode')`. Full dark mode support with CSS custom property color scales.
- **Rating** — first-class `Option<u8>` field on Asset, persisted in sidecar YAML and SQLite. Extracted from XMP `xmp:Rating` during import (conservative merge on first import, overwrite on re-import). Searchable via CLI `rating:N` (exact) and `rating:N+` (minimum). Displayed as stars in `dam show` output. Editable via `dam edit --rating` / `--clear-rating`, web UI inline star rating with htmx, and `PUT /api/asset/{id}/rating` endpoint (accepts `rating=N` 1-5 or `rating=0` to clear). **XMP write-back**: rating changes are automatically written back to `.xmp` recipe files on disk (string-based read-modify-write preserving all other XMP content). After writing, the file is re-hashed and the recipe's content hash is updated in catalog and sidecar. Offline volumes and missing files are silently skipped. This enables bidirectional sync with tools like CaptureOne.
- **Color label** — first-class `Option<String>` field on Asset, persisted in sidecar YAML and SQLite. 7-color set: Red, Orange, Yellow, Green, Blue, Pink, Purple (CaptureOne superset of Lightroom's 5 colors). Stored as canonical English title-case names; `None` = no label. Extracted from XMP `xmp:Label` during import (conservative merge on first import, overwrite on re-import). Searchable via CLI `label:Red` filter. Displayed in `dam show` output. Editable via `dam edit --label <color>` / `--clear-label`, web UI inline color dot picker (7 dots, click to set/toggle), and `PUT /api/asset/{id}/label` endpoint. Batch operations via `PUT /api/batch/label`. **XMP write-back**: label changes are written back to `.xmp` recipe files on disk. Clearing a label removes the `xmp:Label` attribute/element entirely (unlike rating which uses 0 as sentinel). Web UI shows color dots on browse cards, a label filter row on the browse page, and a label section in the batch toolbar.
- **`refresh`** — re-read metadata from changed sidecar/recipe files: `dam refresh [PATHS...] [--volume <label>] [--asset <id>] [--dry-run] [--media]`. Lightweight alternative to `sync` focused exclusively on recipe metadata. Iterates recipe file locations in the catalog, compares on-disk hash to stored hash, and for changed files re-extracts XMP metadata (keywords, rating, description, color label) and updates catalog + sidecar. Non-XMP recipes get hash updated without metadata extraction. `--media` additionally scans JPEG/TIFF variant files and re-extracts their embedded XMP metadata (keywords, rating, description, label, creator, rights) — useful for retroactively extracting embedded XMP from files imported before the feature existed, or after external tools edit the embedded metadata. Without arguments checks all online volumes; with paths scans recipe files under given paths; `--volume` limits to a specific volume; `--asset` limits to a specific asset. `--dry-run` reports changes without applying. Supports `--json`, `--log`, `--time` flags.
- **`saved-search`** (alias `ss`) — manage saved searches (smart albums): named queries stored in `searches.toml` at the catalog root. `dam ss save <NAME> <QUERY> [--sort <SORT>]` saves/replaces a search, `dam ss list` lists all with query and sort, `dam ss run <NAME> [--format <FMT>]` executes and displays results using the same output formats as `dam search`, `dam ss delete <NAME>` removes. Supports `--json`. Web UI: clickable chips on the browse page load saved searches into filter dropdowns. Save button captures the current search state (query text, tag, type, format, rating, label, sort). Chips show rename (…) and delete (×) buttons on hover. API: `GET/POST /api/saved-searches`, `DELETE /api/saved-searches/{name}`.
- **`collection`** (alias `col`) — manage collections (static albums): manually curated lists of asset IDs, backed by SQLite tables (`collections`, `collection_assets`) for fast queries plus `collections.yaml` at catalog root for persistence across `rebuild-catalog`. `dam col create <NAME> [--description <TEXT>]`, `dam col list`, `dam col show <NAME> [--format <FMT>]`, `dam col add <NAME> <ASSET_IDS...>`, `dam col remove <NAME> <ASSET_IDS...>`, `dam col delete <NAME>`. Stdin piping supported for `add`: `dam search -q "rating:5" | xargs dam col add "Best"`. Search filter: `collection:<name>` restricts results to collection members. Web UI: `/collections` page lists all collections with asset counts and a "+ New Collection" button (prompts for name and optional description). Asset detail page shows collection membership chips with × remove buttons. Browse filter row includes a collection dropdown (like type/format/volume) for direct collection filtering, composable with all other search criteria. Browse batch toolbar includes a collection dropdown with context-sensitive buttons: "+ Collection" when no collection is active in the filter dropdown, "− Collection" when a collection is selected (batch dropdown auto-syncs from the filter-row dropdown). Dropdown includes a "New..." option that creates a collection inline via prompt. Collections are restored from YAML during `rebuild-catalog`. API: `GET/POST /api/collections`, `POST/DELETE /api/batch/collection`.
- **Quoted filter values** — search query parser supports double-quoted values for multi-word filters: `tag:"Fools Theater"`, `camera:"Canon EOS R5"`, `lens:"RF 50mm f/1.2"`, `collection:"My Favorites"`, `path:"Photos/Family Trip"`. Unquoted single-word values continue to work as before.
- **Path prefix filter** — `path:<prefix>` restricts search results to assets with a `file_locations.relative_path` starting with the given prefix (SQL `LIKE prefix%`). Useful for finding all assets imported from a specific directory or shoot folder (e.g., `path:Capture/2026-02-22`). Path values are automatically normalized in the CLI: `~` expands to `$HOME`, `./` and `../` resolve relative to the current working directory, and absolute paths matching a registered volume's mount point are stripped to volume-relative with the volume filter implicitly applied (e.g., `path:/Volumes/Photos/Capture/2026` becomes `path:Capture/2026` scoped to that volume). An explicit `volume:` filter takes precedence over the implicit one. Plain relative paths (no `./` prefix) remain volume-relative prefix matches. Composes with all other filters. Available in CLI search, web UI (dedicated text input in browse filter row), and saved searches.
- **`fix-roles`** — fix variant roles in multi-variant assets containing RAW files: `dam fix-roles [PATHS...] [--volume <label>] [--asset <id>] [--apply]`. Scans assets with 2+ variants where at least one is a RAW file. Non-RAW variants with role `Original` are re-roled to `Export`. Assets with only non-RAW variants are untouched (no clear master to determine roles). Dry-run by default (reports what would change); `--apply` writes changes to both sidecar YAML and SQLite catalog. Supports `--json`, `--log`, `--time`. Useful for correcting roles on assets imported before auto-grouping role assignment was fixed.
- **Configuration** (`dam.toml`) — all settings live in `dam.toml` at the catalog root (created by `dam init`). Every section and field is optional; missing values use defaults. Sections: `[preview]` (max_edge, format, quality), `[serve]` (port, bind), `[import]` (exclude patterns, auto_tags), `[dedup]` (prefer). Top-level `default_volume` sets the fallback volume for import. CLI flags override the corresponding `dam.toml` values.
- **`stack`** (alias `st`) — manage stacks (scene grouping): lightweight anonymous groups of assets that collapse in the browse grid to show only the "pick" image with a count badge. One stack per asset, position-ordered (index 0 = pick), auto-dissolve when ≤1 member remains. `dam stack create <IDS...>` creates a stack, `dam stack add <REF> <IDS...>` adds to an existing stack, `dam stack remove <IDS...>` removes from stacks, `dam stack pick <ID>` sets the stack pick, `dam stack dissolve <ID>` dissolves the entire stack, `dam stack list` lists all stacks, `dam stack show <ID>` shows stack members. Stacks persist in `stacks.yaml` at the catalog root and are restored during `rebuild-catalog`. Search filter: `stacked:true` / `stacked:false`. Web UI: stack badge on browse cards, colored left border per stack for visual grouping, collapse/expand toggle in results bar, Stack/Unstack batch buttons, asset detail stack members section with set-pick and dissolve actions. Calendar heatmap respects collapse state. All subcommands support `--json`.
- **Hierarchical tags** — tags can contain `/` as a hierarchy separator (e.g. `animals/birds/eagles`). Searching `tag:animals` matches all descendants. The tags page shows a collapsible tree view with own-count and total-count columns. Interoperates with Lightroom's `lr:hierarchicalSubject` XMP field: hierarchical subjects are imported, merged with flat `dc:subject` tags (components deduplicated), and written back on change. Internally stored with `|` separator to avoid conflicts with literal `/` in tag names.
- **Faceted browse sidebar** — toggleable read-only sidebar on the browse page showing statistical distribution of current results by rating (with bar chart), color label (with color dots), format, volume, tags (top 30), year (with bar chart), and geotagged asset count. Each section is collapsible with state persisted in the browser. Toggle via results bar button or `f` keyboard shortcut. Sidebar preference persisted in localStorage. Hidden on viewports <768px. Backed by `GET /api/facets` endpoint running 8 aggregate queries reusing `build_search_where()` for filter consistency. Counts update automatically on every search change.
